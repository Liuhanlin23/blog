什么时候需要 不需要创建索引

分类于 MySQL ， 索引

一.什么时候需要创建索引

  • 唯一性字段：如商品编码、身份证号等。保证唯一、方便定位。
  • 常用于 WHERE 的字段：提高查询过滤效率；若有多个条件，可建联合索引。
  • 常用于 GROUP BY / ORDER BY 的字段：索引中的数据有序，可避免额外排序，提升分组与排序性能。

理解为什么 GROUP BY / ORDER BY 字段使用索引可以避免额外排序

MySQL 的 B+Tree 索引 是有序结构，索引中的键值天然按升序（或定义顺序）存储。因此：

  • 如果查询排序字段或分组字段使用了索引，MySQL 可直接按照索引顺序扫描数据。
  • 无需再在内存中执行排序操作，性能更高。

三种典型情况

情况一：单列索引可直接排序

若 age 有索引，索引中的记录已按 age 升序存放。
MySQL 直接顺序扫描即可，无需再次排序。

✅ 无需额外排序。

情况二：WHERE 和 ORDER BY 使用不同字段

WHERE 使用 age 索引过滤，但 name 并非同一索引顺序，
筛选结果仍需在内存中重新排序。

⚠️ 需要额外排序（filesort）。

情况三：联合索引同时覆盖过滤与排序

联合索引 (age, name) 既可加速 WHERE，又保持 ORDER BY 顺序一致。

✅ 完全利用索引，一次扫描即可输出有序结果。

总结

  • 索引的有序性使得 GROUP BY / ORDER BY 能直接使用索引顺序。
  • WHERE + ORDER BY 若在同一索引上，可实现“边查边有序”。
  • 不同字段或索引不匹配时，MySQL 必须额外排序，性能下降。

二. 什么时候不需要创建索引

1. 不参与查询条件的字段

  • 在 WHERE、GROUP BY、ORDER BY 中从未出现的字段没必要建索引；
  • 索引只对“查找、排序、分组”起作用，其他字段建了也白搭。

2. 重复值太多的字段

  • 如性别（男女）或状态（0/1），区分度太低；
  • MySQL 查询优化器会自动判断这些情况并放弃索引，直接全表扫描。

3. 表数据太少

  • 数据量几百上千行以内，索引带来的维护开销 > 性能收益。

4. 高频更新的字段

  • 比如用户余额、库存数量等频繁变化字段；
  • 每次更新都要维护 B+Tree，有序性调整会拖慢性能。

一句话总结

不查、不分、不排、重复多、数据少、改太勤 → 都别建索引。

分类于 MySQL ， 日志

三个日志讲完，总结一下update语句的执行过程

查找数据

  • 执行器通过主键 id=1 向存储引擎请求该行记录。
  • 如果数据 在 Buffer Pool 中 → 直接取出；
  • 如果数据 不在 Buffer Pool 中 → 从磁盘读入内存。

执行器判断是否需要更新

  • 对比新旧值：
    • 如果相同 → 不更新；
    • 如果不同 → 把新旧值交给 InnoDB 层执行真正更新。

生成 Undo 日志(开启事务)

  • 更新前，InnoDB 先生成一条 undo log，记录旧值，用于回滚或 MVCC。undo log 会写入 Buffer Pool 中的 Undo 页面，不过在内存修改该 Undo 页面后，需要记录对应的 redo log。

生成 Redo 日志

  • InnoDB 层开始更新记录，会先更新内存（同时标记为脏页），然后将记录写到 redo log 里面，这个时候更新就算完成了(WAL)。后续由后台线程选择一个合适的时机将脏页写入到磁盘。

写入 Binlog

  • 在一条更新语句执行完成后，然后开始记录该语句对应的 binlog，此时记录的 binlog 会被保存到 binlog cache，并没有刷新到硬盘上的 binlog 文件，在事务提交时才会统一将该事务运行过程中的所有 binlog 刷新到硬盘。

两阶段提交（2PC）

  • 事务提交，剩下的就是「两阶段提交」的事情了，接下来就讲这个。

Undo Log（回滚日志）

分类于 MySQL ， 日志

什么是undo log

undo log（回滚日志）：是 InnoDB存储引擎生成的日志， 用来记录数据被修改前“旧值”的一种日志，用于 事务回滚 和 MVCC 多版本并发控制。

注意：

  • undo log（回滚日志）不是一个传统意义上的普通日志文件，不像普通的日志文件那样顺序写入文本，它在 InnoDB 里存放在一个叫 rollback segment 的区域中。
  • 它逻辑上更像一个链表结构（或者更准确地说，一系列链式版本），用于记录每次事务对数据行的修改前的版本。
  • 每条记录通过 roll_pointer（隐藏列之一）指向上一个版本，这样 MySQL 就能从最新版本一路“回滚”找到旧版本。

所以可以总结成一句话：

✅ undo log 不是传统意义上的文件日志，而是存储在表空间中的一系列链式记录，用来追溯数据的历史版本，从而支持事务回滚和 MVCC。

undo log 版本链

当一条数据被修改的时候，InnoDB会

  1. 把修改前的旧值及撤销操作所需的元信息（如操作类型、事务ID等）写入 undo log
  2. 把新值写回数据页
  3. 在数据页中更新两个隐藏字段:
    1. trx_id: 当前修改事务的ID
    2. roll_pointer: 指向刚刚写入的undo log记录

所以每次修改都会生成一个新的undo log 记录，并且数据行会指向这条记录， 这一串 从最新版本到最旧版本 的结构，就是所谓的 版本链

为什么undo log 写入日志的时候不仅要写入旧值，还需要写入旧值变新值操作的原信息

如果 undo log 只是存旧值，那它就变成了“数据快照”或者“历史备份”，
而不是一个能让数据库正确撤销事务的日志。数据库回滚时需要的不仅是“旧值是什么”，
更需要知道，如何将新值变回旧值进行回滚，所以undo log 也记录了回滚操作的原信息

Redo Log 重做日志

分类于 MySQL ， 日志

redo log 是 Inoodb 存储引擎生成的日志， 用来“重做”未写入磁盘数据的日志，用来保证崩溃恢复的 ，实现了事务的持久性，主要用于掉电等故障恢复

它记录的不是“旧版本”，而是 数据页（磁盘的数据页而不是内存页地址）的修改行为（例如“把页X中偏移量Y的值改成Z”）。

为了防止断电导致数据丢失，当有一条记录需要更新的时候，innoDB引擎就会先更新内存（同时标记为脏页），然后将本次对这个页的修改以redo log 的形式记录下来，这个时候更新就算完成了，后续innodb引擎会在适当的时候，由后台线程将缓存在Buffer Pool的脏页刷新到磁盘里，这就是WAL(Write-Ahead Logging)技术，WAL技术指的是，MySQL的写操作并不是立刻写到磁盘上，而是先写日志，然后在合适的时间再写到磁盘上

redo log 要写到磁盘，数据也要写到磁盘，为什么要多次一举

Redo Log 采用*追加写（顺序写）的方式写入磁盘，而数据页更新属于随机写*。
由于磁盘的顺序写效率远高于随机写，Redo Log 能显著降低磁盘 I/O 开销。

类比：顺序写就像在笔记本上按页写字；
随机写则像每写一个字都要翻到对应的页再写，效率显然更低。

产生的 redo log 是直接写入磁盘的吗？

不是，redo log 也是先写入自己的缓存里—— redo log buffer，后续再持久化磁盘

redo log 的刷盘策略

缓存在 redo log buffer 里的 redo log 还是在内存中，它什么时候刷新到磁盘？

主要有下面几个时机：

  • MySQL 正常关闭时；
  • 当 redo log buffer 中记录的写入量大于 redo log buffer 内存空间的一半时，会触发落盘；
  • InnoDB 的后台线程每隔 1 秒，将 redo log buffer 持久化到磁盘。
  • 每次事务提交时都将缓存在 redo log buffer 里的 redo log 直接持久化到磁盘（这个策略可由innodb_flush_log_at_trx_commit 参数控制，下面会说）。

基本流程

当执行更新语句时，InnoDB 会自动开启一个事务：

  • 更新时，生成的 redo log 先写入 redo log buffer（内存）；
  • 事务提交时，再将 redo log buffer 中的内容 顺序写入磁盘文件（redo log 文件）；
  • 这就是默认行为：提交时刷盘（write on commit）。

参数控制：innodb_flush_log_at_trx_commit

InnoDB 提供三种不同的刷盘策略，通过该参数控制，取值为 0、1、2，默认值为 1：

参数值 行为描述 数据安全性 性能
0 事务提交时仅写入 redo log buffer，不主动触发写磁盘操作 ❌ 数据丢失风险高（宕机后 buffer 丢失） ✅ 性能最佳
1（默认） 每次事务提交时，将 redo log buffer 的内容直接写入磁盘 ✅ 最安全（crash-safe） ⚠️ 性能略低
2 每次提交时写入到 redo log 文件（Page Cache），但不强制落盘 ⚠️ 可能丢失 Page Cache 中数据（OS 缓存未刷盘） ✅ 性能较优

💡 注意：
“写入 redo log 文件” ≠ “写入磁盘”。操作系统文件系统通常有 Page Cache（文件缓存），数据只有在 Page Cache 刷盘后才真正持久化。

参数 0 和 2 时 redo log 的刷盘时机

当 innodb_flush_log_at_trx_commit 设置为 0 或 2 时，InnoDB 不会在每次事务提交时立刻把 redo log 写入磁盘，而是由后台线程每隔 1 秒执行一次刷盘操作。

** 参数值 = 0 **

  • *redo log 先写入 redo log buffer（内存）；*
  • 后台线程每 1 秒通过 **write()** 写入操作系统的 Page Cache；
  • 然后调用 **fsync()** 将 Page Cache 内容持久化到磁盘。
  • 若 MySQL 进程崩溃，则 buffer 中的日志会丢失，可能导致最近 1 秒的数据丢失。

** 参数值 = 2 **

  • redo log 在事务提交时写入 操作系统 Page Cache（通过 **write()**）；
  • 后台线程每秒调用 **fsync()** 将 Page Cache 的数据真正写入磁盘；
  • 若 仅 MySQL 崩溃，数据仍然在 OS 缓存中，不会丢失；
  • 只有当 操作系统同时崩溃 才会丢失最近 1 秒的数据。

对比总结

参数值 刷盘时机 崩溃风险 性能 说明
0 每秒后台线程刷盘 MySQL crash → 丢数据 ✅ 最高 数据在内存中
2 每秒后台线程 fsync OS crash → 丢数据 ⚖️ 较高 数据在 OS 缓存中

redo log文件写满会怎么样

**redo log 是一个循环使用的环形日志文件。**当日志写满时，MySQL 会暂停写入，执行 checkpoint，将脏页刷盘，腾出空间后再继续写入。

redo log 的组成

  • InnoDB 默认有一个 重做日志文件组（redo log group），通常包含两个文件：
    ib_logfile0 和 ib_logfile1。
  • 每个文件大小固定，比如每个 1GB，总共 2GB。
  • redo log 文件采用 循环写入（circular writing） 的方式，从头写到尾再回到开头。

写入机制

  1. InnoDB 先写 ib_logfile0；
  2. 当 ib_logfile0 写满时，切换到 ib_logfile1；
  3. 当 ib_logfile1 也写满，再切回 ib_logfile0；
  4. 如此循环往复。

这就像一个“环形缓冲区”，始终在两个文件间循环写入。

write pos 和 checkpoint 的含义

  • write pos：表示当前 redo log 写入的位置（即“尾部”）。
  • checkpoint：表示哪些 redo log 日志已经可以覆盖（即“头部”）。

  • write pos → checkpoint 之间的红色部分：
    存放新的更新记录（已经写入但尚未落盘）。
  • checkpoint → write pos 之间的蓝色部分：
    表示已落盘的脏页，这些 redo log 可以被覆盖。

redo log 写满会怎样？

当 write pos追上checkpoint（即红色区域把蓝色吃光）时：

  • 表示 redo log 空间被写满；
  • MySQL 会暂停所有更新操作，****也就是说 MySQL 会被阻塞（因此所以针对并发量大的系统，适当设置 redo log 的文件大小非常重要）；
  • 系统必须先执行 checkpoint：
    • 把 Buffer Pool 中的脏页写入磁盘；
    • 标记哪些 redo log 已经“安全落盘”；
    • 然后移动 checkpoint，释放出旧的 redo log 空间；
  • 空间腾出后，MySQL 才能恢复执行新的更新操作。

binlog

分类于 MySQL ， 日志

binlog（Binary Log） 是 MySQL Server 层 的日志，用于记录所有对数据库的更改操作（比如 INSERT、UPDATE、DELETE）。不记录查询操作，它是“逻辑日志”，记录的是SQL 语句或行的变化，而不是页的物理修改。

为什么有了redo log 还需要binlog

redo log 是“InnoDB 专属的”

  • redo log 是 InnoDB 存储引擎 自己写的日志；
  • 它只关心磁盘页的物理修改；
  • 目的是为了 崩溃恢复（crash-safe）；
  • 如果数据库突然掉电，InnoDB 可以根据 redo log 把未刷盘的页恢复出来。

所以 redo log 是 “保证事务的持久性” 的。

binlog 是“MySQL Server 层的通用日志”

  • 它是 MySQL 整个系统层面的日志；
  • 所有存储引擎（不仅是 InnoDB）都会写；
  • 记录的是 逻辑操作（SQL 或行变化）；
  • 用来做：
    1. 主从复制（replication）；
    2. 数据恢复（PITR）。

🧩 binlog 是 “保证数据可复制、可恢复” 的关键。

为什么不能用一个？

场景 只靠 redo log 只靠 binlog
系统崩溃 ✅ 可恢复事务 ❌ 因为它只在 commit 时写入，可能丢事务
主从复制 ❌ redo log 是物理的，主库没法发给从库 ✅ 复制用 binlog
跨存储引擎 ❌ redo 仅 InnoDB 可用 ✅ binlog 所有引擎都能写
写入效率 高（循环写） 低（追加写）

binlog 的三种格式

① STATEMENT（语句模式）

  • 每条修改数据的 SQL 语句 都会被记录到 binlog 中。
  • 主从复制时，从库会重新执行同样的 SQL 语句。
  • 优点：日志量小、效率高。
  • 缺点：如果 SQL 中使用了 随机或时间函数（如 **UUID()**、**NOW()**），主从库结果可能不一致。

② ROW（行模式）

  • 记录的是每一行数据修改前后的变化结果。
  • 不会出现动态函数导致的不一致问题。
  • 优点：复制更安全、主从数据完全一致。
  • 缺点：更新多行会生成多条记录，binlog 文件体积更大。

③ MIXED（混合模式）

  • 结合前两种：
    • 一般语句用 STATEMENT 模式；
    • 遇到不安全语句（如包含随机函数）自动切换为 ROW 模式。
  • 优点：兼顾安全与效率。

📘 一句话记忆：

  • STATEMENT：记录 SQL 语句 → 小但可能不一致
  • ROW：记录行变化 → 安全但体积大
  • MIXED：自动选择 → 平衡两者

binlog和redo log的写入方式

  • binlog 是追加写，写满一个文件，就创建一个新的文件继续写，不会覆盖以前的日志，保存的是全量的日志。
  • redo log 是循环写，日志空间大小是固定，全部写满就从头开始，保存未被刷入磁盘的脏页日志。

如果不小心将整个数据库的数据删除了，使用binlog还是redo log 文件恢复数据?

使用binlog ，因为 redo log 文件是循环写，是会边写边擦除日志的，只记录未被刷入磁盘的数据的物理日志，已经刷入磁盘的数据都会从 redo log 文件里擦除。

binlog 文件保存的是全量的日志，也就是保存了所有数据变更的情况，理论上只要记录在 binlog 上的数据，都可以恢复，所以如果不小心整个数据库的数据被删除了，得用 binlog 文件恢复数据。

binlog 什么时候刷盘

binlog 的写入流程

1️⃣ 事务执行阶段

  • SQL 执行时，日志内容先写入 binlog cache（每个线程独立），还没落盘。

2️⃣ 事务提交时（commit）

  • 把当前事务对应的 binlog cache 内容，一次性写入 binlog 文件（write）；
  • 然后根据配置决定是否执行 fsync（真正落盘）；
  • 最后清空线程的 binlog cache。

虽然每个线程有自己 binlog cache，但是最终都写到同一个 binlog 文件：

sync_binlog 参数控制刷盘频率

参数值 刷盘策略 性能 风险
sync_binlog = 0 每次事务提交只 write，不 fsync，由操作系统决定何时落盘 性能最好 崩溃可能丢日志
sync_binlog = 1 每次事务提交后立刻 fsync 最安全 性能最差
sync_binlog = N (N>1) 每提交 N 个事务后执行一次 fsync 性能与安全折中 最多丢 N 个事务日志

MySQL 默认值与推荐设置

  • 默认：sync_binlog = 0（性能优先，风险较高）
  • 推荐：根据业务情况选择
  • 若要求高可靠性（金融类） → sync_binlog = 1
  • 若能容忍少量数据丢失 → sync_binlog = 100~1000

标量子查询

分类于 MySQL ， SQL

标量子查询是 只返回一个值（一个单元格） 的子查询。
这个值可以出现在主查询的 SELECT、WHERE、HAVING 等语句中，就像一个普通字段或常量一样使用。

换句话说：
👉 标量子查询的结果相当于“先执行一个查询得到一个数字/字符串”，
然后把它当成主查询的一部分。

注意事项

1. 标量子查询必须只返回 一行一列 否则会报错

比如： SELECT (SELECT dept_name FROM dept) – 如果返回多行就报错

标量子查询（scalar subquery）可以理解为：

“在外层查询运行时，先执行里面的小查询，把结果当作一个值代入外层”。

2️⃣ 对每一行，执行内层子查询：SELECT dept_name FROM dept WHERE dept.id = emp.dept_id;

如果是“相关子查询”，每行执行一次 （性能低，慎用，除非条件固定，结果稳定）

✅ 子查询结果缓存（Subquery Cache）
当子查询的输入条件相同且结果未过期时，数据库可以复用上一次的查询结果，而无需重新计算，从而提高查询性能。
👉 因此，在子查询**条件固定、结果稳定的场景下，使用子查询通常性能较高，值得推荐**。

内连接

内连接（INNER JOIN）本质上就是笛卡尔积，只不过通常会加上连接条件。
➤ 写了条件 ⇒ 有条件的笛卡尔积（只保留匹配行）
➤ 没写条件 ⇒ 无条件的笛卡尔积（所有组合都出现）

SQL语句

SELECT 列名列表 FROM 表1 [AS] 别名1 （INNER） JOIN 表2 [AS] 别名2 ON 连接条件;

INNER关键字可省略

左连接，右连接

  • LEFT JOIN： 保留左表全部记录，右表匹配不到的部分字段用 **NULL** 填充。
  • RIGHT JOIN： 保留右表全部记录，左表匹配不到的部分字段用 **NULL** 填充。
  • ON： 指定两表的关联条件（例如 s.class_id = c.id），用于判断哪些行是匹配的。

SQL语句

SELECT 列名列表 FROM 表1 [AS] 别名1 LEFT JOIN 表2 [AS] 别名2 ON 连接条件;

SELECT 列名列表 FROM 表1 [AS] 别名1 RIGHT JOIN 表2 [AS] 别名2 ON 连接条件;

全连接

全连接（FULL JOIN / FULL OUTER JOIN）
会返回左表和右表的所有记录，

  • 匹配的行：正常合并
  • 不匹配的行：用 NULL 填充缺失部分

换句话说：

FULL JOIN = LEFT JOIN + RIGHT JOIN 的并集

SQL语句

SELECT 列名列表 FROM 表1 [AS] 别名1 FULL JOIN 表2 [AS] 别名2 ON 连接条件;

自连接

自连接（Self Join）
是指一张表自己连接自己，用于查询同一张表中存在层级关系或关联关系的数据。

本质上就是：

把同一张表当作两张表来用，通过别名区分。

SQL语句

SELECT 列名列表 FROM 表名 AS 别名1 [INNER | LEFT | RIGHT] JOIN 表名 AS 别名2 ON 连接条件;

✅ 注意：

  • 必须给表起两个不同的别名（例如 a 和 b）；
  • 否则 MySQL 无法区分同一张表的两次引用。

联合查询

联合查询（UNION）
用于将多个 **SELECT** 查询的结果纵向合并为一个结果集。

它不是“横向连接”（那是 JOIN），
而是上下拼接，把多次查询的结果叠加在一起。

SQL语句

SELECT 列名列表 FROM 表1 … UNION [ALL]

SELECT 列名列表 FROM 表2 …;

UNION 和UNION ALL的区别

UNION 会自动去重，UNION ALL 不会去重。

分组查询

分类于 MySQL ， SQL

MySQL 的 分组查询 主要是通过 GROUP BY 子句来实现的，它将查询结果按指定的列进行分组，通常配合聚合函数（如 COUNT()、SUM()、AVG()、MAX()、MIN() 等）来对每一组数据进行计算和统计。

基本语法

SELECT column1, column2, aggregation_function(column3)

FROM table_name

GROUP BY column1, column2;


  1. **SELECT** 用于指定查询的列，包含：
  • 普通列：如 column1 和 column2，用于分组的依据。
  • 聚合函数：如 aggregation_function(column3)，对某列（例如 column3）应用聚合操作。

2. **GROUP BY**``column1, column2：按 column1 和 column2 列分组，将数据分成多个组，每个组的值在这两列上是相同的。

3. ``aggregation_function(column3)：对列 column3 的数据执行聚合操作并返回一个结果。常见的聚合函数有：

  • COUNT(column3)：返回 column3 中非 NULL 值的行数。
  • SUM(column3)：返回 column3 所有值的和。
  • AVG(column3)：返回 column3 所有值的平均值。
  • MIN(column3)：返回 column3 中的最小值。
  • MAX(column3)：返回 column3 中的最大值。
  1. 注意

在 GROUP BY 分组查询中，聚合函数中的列不一定需要出现在 <font style="color:#DF2A3F;">GROUP BY</font> 中，但如果列没有出现在 <font style="color:#DF2A3F;">GROUP BY</font> 中且没有经过聚合函数处理，它不能直接在 <font style="color:#DF2A3F;">SELECT</font> 中查询，否则查询会报错。

例子

假设有一个 employees 表，包含 department_id（部门ID）、salary（薪资）、hire_date（入职日期）等列。

按部门分组，计算每个部门的总薪资

SELECT department_id, SUM(salary) AS total_salary

FROM employees

GROUP BY department_id;

  • 这里，department_id 是分组依据，SUM(salary) 计算每个部门的总薪资。
  • salary 并没有出现在 GROUP BY 中，但它通过 SUM() 聚合函数来处理。

按部门分组，计算每个部门的最高薪资和入职最早的员工

SELECT department_id, MAX(salary) AS highest_salary, MIN(hire_date) AS earliest_hire_date

FROM employees

GROUP BY department_id;

  • department_id 是分组依据，MAX(salary) 和 MIN(hire_date) 分别用于计算每个部门的最高薪资和最早入职的日期。
  • salary 和 hire_date 并没有出现在 GROUP BY 中，但它们通过聚合函数来处理。

错误示例（如果你没有使用聚合函数处理其他列）

SELECT department_id, hire_date

FROM employees

GROUP BY department_id;

这个查询会报错，因为 hire_date 没有使用聚合函数，且它也没有出现在 GROUP BY 中。在这种情况下，数据库不能确定如何处理 hire_date，因为每个部门可能有多个不同的入职日期。

HAVING 子句

HAVING 子句的作用

  • HAVING 用于对 分组后的数据 进行过滤。它类似于 WHERE 子句，但不同的是，WHERE 用于过滤 原始数据，而 HAVING 用于过滤 已经分组 后的结果。
  • **HAVING** 常与聚合函数一起使用，用于根据聚合结果进行条件判断。

HAVING 子句的语法

SELECT column1, column2, aggregation_function(column3)

FROM table_name

GROUP BY column1, column2

HAVING condition;


  • **HAVING** 子句在分组之后应用，用来过滤聚合结果。
  • HAVING 后面可以跟聚合函数的条件（例如 COUNT()、SUM()、AVG() 等）。
  • WHERE 不能用于过滤聚合后的结果，必须先使用 GROUP BY 进行分组，再使用 HAVING 来过滤。

HAVING 和 WHERE 的区别：

  • WHERE 用于过滤 原始数据，不能与聚合函数一起使用。
  • HAVING 用于过滤 分组后的数据，可以与聚合函数一起使用。

分组查询数据链路

假设我们有以下 employees 表：

department_id employee_id salary
1 101 5000
1 102 6000
2 103 7000
2 104 8000
3 105 9000
3 106 10000

我们要执行一个按 department_id 分组的查询，来计算每个部门的 总薪资（SUM(salary)）和 员工人数（COUNT(employee_id)），并筛选出薪资总和大于 15000 的部门。

SELECT department_id, SUM(salary) AS total_salary, COUNT(employee_id) AS employee_count

FROM employees

GROUP BY department_id

HAVING SUM(salary) > 15000;


1. 原始数据：

我们从表 employees 中根据where查询出原始数据

2. 分组操作：

根据 department_id 列将数据分成三个组。

  • 组 1: department_id = 1
    • (101, 5000), (102, 6000)
  • 组 2: department_id = 2
    • (103, 7000), (104, 8000)
  • 组 3: department_id = 3
    • (105, 9000), (106, 10000)

3. 应用聚合函数：

对每个分组应用聚合函数来计算 总薪资（SUM(salary)）和 员工数（COUNT(employee_id)）。

  • 组 1:
    • 总薪资： 5000 + 6000 = 11000
    • 员工数： COUNT(2)
  • 组 2:
    • 总薪资： 7000 + 8000 = 15000
    • 员工数： COUNT(2)
  • 组 3:
    • 总薪资： 9000 + 10000 = 19000
    • 员工数： COUNT(2)

4. 查询结果：

这时，每个分组的聚合结果已被计算出来。

department_id total_salary employee_count
1 11000 2
2 15000 2
3 19000 2

5. 应用 **HAVING** 子句：

HAVING 用于过滤分组后的结果。在这个查询中，我们只保留总薪资大于 15000 的部门。

department_id total_salary employee_count
3 19000 2

6. 最终结果：

最终，查询结果只包含总薪资大于 15000 的部门：

department_id total_salary employee_count
3 19000 2

ROW NUMBER

分类于 MySQL ， SQL

**ROW_NUMBER()** 是 SQL 中的窗口函数之一，用于为结果集中的每一行分配一个唯一的序列号。这个序列号是根据你指定的排序规则（通过 ORDER BY 子句）生成的，通常用于分页、排序、去重等场景。

语法：

ROW_NUMBER() OVER (PARTITION BY partition_column ORDER BY order_column)


  • **PARTITION BY**（可选）：用于指定分组。ROW_NUMBER() 会在每个分组内重新计数。如果没有指定 PARTITION BY，则表示整个结果集为一个分组。
  • **ORDER BY**：指定序列号的排序方式。ROW_NUMBER() 会根据这个列的顺序为每行分配一个唯一的序列号。

返回值：

**ROW_NUMBER()** 返回一个整数，从 1 开始，为每一行生成一个唯一的、按指定排序的序列号。

示例：

假设有一个 employees 表，包含以下数据：

emp_no name salary
101 Alice 5000
102 Bob 6000
103 Charlie 7000
104 David 8000
105 Eva 9000

我们想要为这个结果集按 salary 排序，并为每个员工生成一个序列号。

  1. **按 **salary** 排序使用 ****ROW_NUMBER()**：

SELECT emp_no, name, salary,

   ROW_NUMBER() OVER (ORDER BY salary DESC) AS row_num

FROM employees;


结果：

emp_no name salary row_num
105 Eva 9000 1
104 David 8000 2
103 Charlie 7000 3
102 Bob 6000 4
101 Alice 5000 5

ROW_NUMBER() 按照 salary 列从高到低排序，并为每行分配一个唯一的序列号。

  1. 使用 **PARTITION BY** 分组后进行 **ROW_NUMBER()** 排序：

假设我们对 department_id 进行分组，并在每个部门内按 salary 排序，为每个部门的员工生成一个序列号。

SELECT emp_no, name, department_id, salary,

   ROW_NUMBER() OVER (PARTITION BY department_id ORDER BY salary DESC) AS row_num

FROM employees;


结果：

假设数据如下，带有 department_id：

emp_no name department_id salary
101 Alice 1 5000
102 Bob 1 6000
103 Charlie 2 7000
104 David 2 8000
105 Eva 2 9000

执行结果：

emp_no name department_id salary row_num
102 Bob 1 6000 1
101 Alice 1 5000 2
105 Eva 2 9000 1
104 David 2 8000 2
103 Charlie 2 7000 3

在每个 department_id 中，ROW_NUMBER() 从 1 开始，按 salary 降序排列

应用查询

分页查询：

ROW_NUMBER() 常用于分页查询，结合 WHERE 子句和 BETWEEN 子句来获取特定范围的行。

WITH NumberedEmployees AS (

SELECT emp_no, name, salary, 

       ROW_NUMBER() OVER (ORDER BY salary DESC) AS row_num

FROM employees

)

SELECT emp_no, name, salary

FROM NumberedEmployees

WHERE row_num BETWEEN 10 AND 20;


去重

当我们想去除某些重复行时，ROW_NUMBER() 可以帮助我们标记重复的行，并只保留特定的行。

按特定顺序排序并标记每一行的顺序

比如，当我们需要按某种规则排序并标记每行的排名时，ROW_NUMBER() 很有用。

总结

  • **ROW_NUMBER()** 是一个窗口函数，用于根据特定排序为结果集中的每一行分配一个唯一的序列号。
  • 可以通过 **PARTITION BY** 分组，并使用 **ORDER BY** 进行排序。
  • 它广泛应用于分页查询、去重、排序等场景。

连接池

分类于 MySQL
  1. 在springboot+mybatis 场景中，默认连接池是HikariCP（spring boot 默认DataSource），Mybatis 只是复用它
  2. 连接池的优点
  • ** 复用连接，降低开销**：避免频繁 TCP/TLS 握手 + 认证，显著降低延迟。
  • 吞吐更稳：提前或按需维护一组可用连接，减少高峰期建连抖动。
  • 超时与容错统一：连接获取超时、校验超时、泄漏检测等集中可配，运维友好。

Mysql 的连接超时机制

  • MySQL 定义了空闲连接的最大空闲时长,由 wait_timeout 参数控制,默认八小时,如果空闲连接超过了这个时间,连接器就会自动将它断开
  • 一个处于空闲状态的连接被服务端主动断开后，这个客户端并不会马上知道，等到客户端在发起下一个请求的时候，才会收到这样的报错“ERROR 2013 (HY000): Lost connection to MySQL server during query”。

同时也要注意的是对于springboot 的默认连接池，如果在执行sql 的时候发现连接已经被sql 连接器中断，那么就会导致这个sql 执行报错不会重新连接

如何解决这个问题

  1. 可以在连接池中设置先探活,死了就丢掉重新连接
  2. 在MySql 中断连接之前,连接池主动下线旧连接

注: 关于连接池主动下线旧连接会导致连接越来越少么?

	不会 连接池内配置了一个最大 ，最小连接数量,维护了一个区间

MySQL 缓存机制

在MySQL 中 有两种缓存，一种是server 层的缓存，一种是存储引擎层的缓存Buffer Pool

  1. server 层的缓存: MySQL 收到SQL语句后会解析是不是查询,如果是查询就先去查询缓存，查询缓存是以key-value 形式保存在内存中的，key 保存的是查询语句,value 是结果集,但是这个server 层的缓存很鸡肋,因为如果只要有一个表的数据更新了,那么这个表的所有缓存都会被删掉，所以MySQL 8.0之后删除了server 层缓存,之前的版本可以通过配置关掉
  2. buffer pool 缓存： 缓存的是数据页/索引页（物理页），用来减少磁盘 I/O。
