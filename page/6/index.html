Hset 底层数据结构是由listpack或hash表实现的,当数据较少时并且所有数据数据量少于64字节(默认值可配置)采用listpack ,数据较大时采用hash来实现

Redis 怎么用它表示一个hash

很巧妙：Redis 把每个 key-value 对依次塞进 listpack 里，也就是：

[key1, value1, key2, value2, key3, value3, …]

换句话说：

  • 奇数位置是 key；
  • 偶数位置是 value。

这就是 Redis 的“小型哈希”在内存中的真实样子。

查找时 Redis 会顺序扫描（O(N)），但因为数据很小，这样反而更快、更省内存。

这样设计的优点

特点 listpack 编码（小hash） hashtable 编码（大hash）
内存占用 极小（紧凑连续） 较大（多层结构）
查找复杂度 O(N) O(1)
场景 小、短、频繁创建销毁的哈希对象 大、长期存在的哈希对象

MySQL 缓存机制

在MySQL 中 有两种缓存，一种是server 层的缓存，一种是存储引擎层的缓存Buffer Pool

  1. server 层的缓存: MySQL 收到SQL语句后会解析是不是查询,如果是查询就先去查询缓存，查询缓存是以key-value 形式保存在内存中的，key 保存的是查询语句,value 是结果集,但是这个server 层的缓存很鸡肋,因为如果只要有一个表的数据更新了,那么这个表的所有缓存都会被删掉，所以MySQL 8.0之后删除了server 层缓存,之前的版本可以通过配置关掉
  2. buffer pool 缓存： 缓存的是数据页/索引页（物理页），用来减少磁盘 I/O。

主键索引自增

1. InnoDB 的主键是聚簇索引

  • 数据行按主键顺序存储在 B+Tree 叶子节点中。
  • 主键顺序决定数据的真实物理存储顺序。

一、磁盘与 I/O 的基础概念

磁盘的最小读写单元是 扇区（Sector），通常大小为 512 字节。
操作系统以 页（Page） 为单位读写数据（MySQL 默认页大小为 16KB）。

为了方便理解，这里以扇区为例模拟磁盘读写。

二、没有索引时：全表扫描

假设：

  • 每行数据大小为 128 字节；
  • 一张表有 800 行数据；
  • 磁盘每个扇区大小为 512 字节。

总数据量为： 800 × 128 = 102,400 字节

每个扇区能存放：512 ÷ 128 = 4 行数据

因此需要： 800 ÷ 4 = 200 个扇区

如果没有索引，MySQL 查询时必须扫描全部 200 个扇区，也就是 200 次磁盘 I/O。

三、引入索引：减少磁盘 I/O

键值（key） → 数据所在磁盘页的地址（pointer）

所以此时就就将200次磁盘io变为了现在索引中查找100*128/512 在索引中找的的数据进行查找

同样的我们可以在进行二级索引,三级索引…减少磁盘io

四、多级索引：B+Tree 结构的出现

当索引记录本身也很大时（例如几百万条），
我们还可以在索引上再建索引 —— 即 多级索引。
这就形成了 MySQL 的 B+Tree 索引结构。

每一层索引都存储下一层节点的磁盘地址。
通过分层目录结构，MySQL 可以在 3～4 次磁盘 I/O 内定位到目标数据

五、B+Tree 查找过程示例

查找步骤：

  1. 在根节点中找到 33 < 37 < 65，进入对应的下层节点；
  2. 在第二层节点中锁定包含 37 的区间；
  3. 到叶子节点中精确查找 id=37；
  4. 如果是 聚簇索引，叶子节点直接存放数据；
    如果是 二级索引，叶子节点存放主键值，再回表查找数据。

🔹 访问树的每一层都需要一次磁盘 I/O；
通常 B+Tree 高度为 3～4 层，
所以 MySQL 只需 3～4 次 I/O 就能定位到任意记录。

三个日志讲完，总结一下update语句的执行过程

查找数据

  • 执行器通过主键 id=1 向存储引擎请求该行记录。
  • 如果数据 在 Buffer Pool 中 → 直接取出；
  • 如果数据 不在 Buffer Pool 中 → 从磁盘读入内存。

执行器判断是否需要更新

  • 对比新旧值：
    • 如果相同 → 不更新；
    • 如果不同 → 把新旧值交给 InnoDB 层执行真正更新。

生成 Undo 日志(开启事务)

  • 更新前，InnoDB 先生成一条 undo log，记录旧值，用于回滚或 MVCC。undo log 会写入 Buffer Pool 中的 Undo 页面，不过在内存修改该 Undo 页面后，需要记录对应的 redo log。

生成 Redo 日志

  • InnoDB 层开始更新记录，会先更新内存（同时标记为脏页），然后将记录写到 redo log 里面，这个时候更新就算完成了(WAL)。后续由后台线程选择一个合适的时机将脏页写入到磁盘。

写入 Binlog

  • 在一条更新语句执行完成后，然后开始记录该语句对应的 binlog，此时记录的 binlog 会被保存到 binlog cache，并没有刷新到硬盘上的 binlog 文件，在事务提交时才会统一将该事务运行过程中的所有 binlog 刷新到硬盘。

两阶段提交（2PC）

  • 事务提交，剩下的就是「两阶段提交」的事情了，接下来就讲这个。

什么是undo log

undo log（回滚日志）：是 InnoDB存储引擎生成的日志， 用来记录数据被修改前“旧值”的一种日志，用于 事务回滚 和 MVCC 多版本并发控制。

举个例子：

注意：

  • undo log（回滚日志）不是一个传统意义上的普通日志文件，不像普通的日志文件那样顺序写入文本，它在 InnoDB 里存放在一个叫 rollback segment 的区域中。
  • 它逻辑上更像一个链表结构（或者更准确地说，一系列链式版本），用于记录每次事务对数据行的修改前的版本。
  • 每条记录通过 roll_pointer（隐藏列之一）指向上一个版本，这样 MySQL 就能从最新版本一路“回滚”找到旧版本。

所以可以总结成一句话：

✅ undo log 不是传统意义上的文件日志，而是存储在表空间中的一系列链式记录，用来追溯数据的历史版本，从而支持事务回滚和 MVCC。

undo log 版本链

当一条数据被修改的时候，InnoDB会

  1. 把修改前的旧值及撤销操作所需的元信息（如操作类型、事务ID等）写入 undo log
  2. 把新值写回数据页
  3. 在数据页中更新两个隐藏字段:
    1. trx_id: 当前修改事务的ID
    2. roll_pointer: 指向刚刚写入的undo log记录

所以每次修改都会生成一个新的undo log 记录，并且数据行会指向这条记录， 这一串 从最新版本到最旧版本 的结构，就是所谓的 版本链

为什么undo log 写入日志的时候不仅要写入旧值，还需要写入旧值变新值操作的原信息

如果 undo log 只是存旧值，那它就变成了“数据快照”或者“历史备份”，
而不是一个能让数据库正确撤销事务的日志。数据库回滚时需要的不仅是“旧值是什么”，
更需要知道，如何将新值变回旧值进行回滚，所以undo log 也记录了回滚操作的原信息

redo log 是 Inoodb 存储引擎生成的日志， 用来“重做”未写入磁盘数据的日志，用来保证崩溃恢复的 ，实现了事务的持久性，主要用于掉电等故障恢复

它记录的不是“旧版本”，而是 数据页（磁盘的数据页而不是内存页地址）的修改行为（例如“把页X中偏移量Y的值改成Z”）。

为了防止断电导致数据丢失，当有一条记录需要更新的时候，innoDB引擎就会先更新内存（同时标记为脏页），然后将本次对这个页的修改以redo log 的形式记录下来，这个时候更新就算完成了，后续innodb引擎会在适当的时候，由后台线程将缓存在Buffer Pool的脏页刷新到磁盘里，这就是WAL(Write-Ahead Logging)技术，WAL技术指的是，MySQL的写操作并不是立刻写到磁盘上，而是先写日志，然后在合适的时间再写到磁盘上

redo log 要写到磁盘，数据也要写到磁盘，为什么要多次一举

Redo Log 采用*追加写（顺序写）的方式写入磁盘，而数据页更新属于随机写*。
由于磁盘的顺序写效率远高于随机写，Redo Log 能显著降低磁盘 I/O 开销。

类比：顺序写就像在笔记本上按页写字；
随机写则像每写一个字都要翻到对应的页再写，效率显然更低。

产生的 redo log 是直接写入磁盘的吗？

不是，redo log 也是先写入自己的缓存里—— redo log buffer，后续再持久化磁盘

redo log 的刷盘策略

缓存在 redo log buffer 里的 redo log 还是在内存中，它什么时候刷新到磁盘？

主要有下面几个时机：

  • MySQL 正常关闭时；
  • 当 redo log buffer 中记录的写入量大于 redo log buffer 内存空间的一半时，会触发落盘；
  • InnoDB 的后台线程每隔 1 秒，将 redo log buffer 持久化到磁盘。
  • 每次事务提交时都将缓存在 redo log buffer 里的 redo log 直接持久化到磁盘（这个策略可由innodb_flush_log_at_trx_commit 参数控制，下面会说）。

基本流程

当执行更新语句时，InnoDB 会自动开启一个事务：

  • 更新时，生成的 redo log 先写入 redo log buffer（内存）；
  • 事务提交时，再将 redo log buffer 中的内容 顺序写入磁盘文件（redo log 文件）；
  • 这就是默认行为：提交时刷盘（write on commit）。

参数控制：innodb_flush_log_at_trx_commit

InnoDB 提供三种不同的刷盘策略，通过该参数控制，取值为 0、1、2，默认值为 1：

参数值 行为描述 数据安全性 性能
0 事务提交时仅写入 redo log buffer，不主动触发写磁盘操作 ❌ 数据丢失风险高（宕机后 buffer 丢失） ✅ 性能最佳
1（默认） 每次事务提交时，将 redo log buffer 的内容直接写入磁盘 ✅ 最安全（crash-safe） ⚠️ 性能略低
2 每次提交时写入到 redo log 文件（Page Cache），但不强制落盘 ⚠️ 可能丢失 Page Cache 中数据（OS 缓存未刷盘） ✅ 性能较优

💡 注意：
“写入 redo log 文件” ≠ “写入磁盘”。操作系统文件系统通常有 Page Cache（文件缓存），数据只有在 Page Cache 刷盘后才真正持久化。

参数 0 和 2 时 redo log 的刷盘时机

当 innodb_flush_log_at_trx_commit 设置为 0 或 2 时，InnoDB 不会在每次事务提交时立刻把 redo log 写入磁盘，而是由后台线程每隔 1 秒执行一次刷盘操作。

** 参数值 = 0 **

  • *redo log 先写入 redo log buffer（内存）；*
  • 后台线程每 1 秒通过 **write()** 写入操作系统的 Page Cache；
  • 然后调用 **fsync()** 将 Page Cache 内容持久化到磁盘。
  • 若 MySQL 进程崩溃，则 buffer 中的日志会丢失，可能导致最近 1 秒的数据丢失。

** 参数值 = 2 **

  • redo log 在事务提交时写入 操作系统 Page Cache（通过 **write()**）；
  • 后台线程每秒调用 **fsync()** 将 Page Cache 的数据真正写入磁盘；
  • 若 仅 MySQL 崩溃，数据仍然在 OS 缓存中，不会丢失；
  • 只有当 操作系统同时崩溃 才会丢失最近 1 秒的数据。

对比总结

参数值 刷盘时机 崩溃风险 性能 说明
0 每秒后台线程刷盘 MySQL crash → 丢数据 ✅ 最高 数据在内存中
2 每秒后台线程 fsync OS crash → 丢数据 ⚖️ 较高 数据在 OS 缓存中

redo log文件写满会怎么样

**redo log 是一个循环使用的环形日志文件。**当日志写满时，MySQL 会暂停写入，执行 checkpoint，将脏页刷盘，腾出空间后再继续写入。

redo log 的组成

  • InnoDB 默认有一个 重做日志文件组（redo log group），通常包含两个文件：
    ib_logfile0 和 ib_logfile1。
  • 每个文件大小固定，比如每个 1GB，总共 2GB。
  • redo log 文件采用 循环写入（circular writing） 的方式，从头写到尾再回到开头。

写入机制

  1. InnoDB 先写 ib_logfile0；
  2. 当 ib_logfile0 写满时，切换到 ib_logfile1；
  3. 当 ib_logfile1 也写满，再切回 ib_logfile0；
  4. 如此循环往复。

这就像一个“环形缓冲区”，始终在两个文件间循环写入。

write pos 和 checkpoint 的含义

  • write pos：表示当前 redo log 写入的位置（即“尾部”）。
  • checkpoint：表示哪些 redo log 日志已经可以覆盖（即“头部”）。

  • write pos → checkpoint 之间的红色部分：
    存放新的更新记录（已经写入但尚未落盘）。
  • checkpoint → write pos 之间的蓝色部分：
    表示已落盘的脏页，这些 redo log 可以被覆盖。

redo log 写满会怎样？

当 write pos追上checkpoint（即红色区域把蓝色吃光）时：

  • 表示 redo log 空间被写满；
  • MySQL 会暂停所有更新操作，****也就是说 MySQL 会被阻塞（因此所以针对并发量大的系统，适当设置 redo log 的文件大小非常重要）；
  • 系统必须先执行 checkpoint：
    • 把 Buffer Pool 中的脏页写入磁盘；
    • 标记哪些 redo log 已经“安全落盘”；
    • 然后移动 checkpoint，释放出旧的 redo log 空间；
  • 空间腾出后，MySQL 才能恢复执行新的更新操作。
