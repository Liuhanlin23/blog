binlog（Binary Log） 是 MySQL Server 层 的日志，用于记录所有对数据库的更改操作（比如 INSERT、UPDATE、DELETE）。不记录查询操作，它是“逻辑日志”，记录的是SQL 语句或行的变化，而不是页的物理修改。

为什么有了redo log 还需要binlog

redo log 是“InnoDB 专属的”

  • redo log 是 InnoDB 存储引擎 自己写的日志；
  • 它只关心磁盘页的物理修改；
  • 目的是为了 崩溃恢复（crash-safe）；
  • 如果数据库突然掉电，InnoDB 可以根据 redo log 把未刷盘的页恢复出来。

所以 redo log 是 “保证事务的持久性” 的。

binlog 是“MySQL Server 层的通用日志”

  • 它是 MySQL 整个系统层面的日志；
  • 所有存储引擎（不仅是 InnoDB）都会写；
  • 记录的是 逻辑操作（SQL 或行变化）；
  • 用来做：
    1. 主从复制（replication）；
    2. 数据恢复（PITR）。

🧩 binlog 是 “保证数据可复制、可恢复” 的关键。

为什么不能用一个？

场景 只靠 redo log 只靠 binlog
系统崩溃 ✅ 可恢复事务 ❌ 因为它只在 commit 时写入，可能丢事务
主从复制 ❌ redo log 是物理的，主库没法发给从库 ✅ 复制用 binlog
跨存储引擎 ❌ redo 仅 InnoDB 可用 ✅ binlog 所有引擎都能写
写入效率 高（循环写） 低（追加写）

binlog 的三种格式

① STATEMENT（语句模式）

  • 每条修改数据的 SQL 语句 都会被记录到 binlog 中。
  • 主从复制时，从库会重新执行同样的 SQL 语句。
  • 优点：日志量小、效率高。
  • 缺点：如果 SQL 中使用了 随机或时间函数（如 **UUID()**、**NOW()**），主从库结果可能不一致。

② ROW（行模式）

  • 记录的是每一行数据修改前后的变化结果。
  • 不会出现动态函数导致的不一致问题。
  • 优点：复制更安全、主从数据完全一致。
  • 缺点：更新多行会生成多条记录，binlog 文件体积更大。

③ MIXED（混合模式）

  • 结合前两种：
    • 一般语句用 STATEMENT 模式；
    • 遇到不安全语句（如包含随机函数）自动切换为 ROW 模式。
  • 优点：兼顾安全与效率。

📘 一句话记忆：

  • STATEMENT：记录 SQL 语句 → 小但可能不一致
  • ROW：记录行变化 → 安全但体积大
  • MIXED：自动选择 → 平衡两者

binlog和redo log的写入方式

  • binlog 是追加写，写满一个文件，就创建一个新的文件继续写，不会覆盖以前的日志，保存的是全量的日志。
  • redo log 是循环写，日志空间大小是固定，全部写满就从头开始，保存未被刷入磁盘的脏页日志。

如果不小心将整个数据库的数据删除了，使用binlog还是redo log 文件恢复数据?

使用binlog ，因为 redo log 文件是循环写，是会边写边擦除日志的，只记录未被刷入磁盘的数据的物理日志，已经刷入磁盘的数据都会从 redo log 文件里擦除。

binlog 文件保存的是全量的日志，也就是保存了所有数据变更的情况，理论上只要记录在 binlog 上的数据，都可以恢复，所以如果不小心整个数据库的数据被删除了，得用 binlog 文件恢复数据。

binlog 什么时候刷盘

binlog 的写入流程

1️⃣ 事务执行阶段

  • SQL 执行时，日志内容先写入 binlog cache（每个线程独立），还没落盘。

2️⃣ 事务提交时（commit）

  • 把当前事务对应的 binlog cache 内容，一次性写入 binlog 文件（write）；
  • 然后根据配置决定是否执行 fsync（真正落盘）；
  • 最后清空线程的 binlog cache。

虽然每个线程有自己 binlog cache，但是最终都写到同一个 binlog 文件：

sync_binlog 参数控制刷盘频率

参数值 刷盘策略 性能 风险
sync_binlog = 0 每次事务提交只 write，不 fsync，由操作系统决定何时落盘 性能最好 崩溃可能丢日志
sync_binlog = 1 每次事务提交后立刻 fsync 最安全 性能最差
sync_binlog = N (N>1) 每提交 N 个事务后执行一次 fsync 性能与安全折中 最多丢 N 个事务日志

MySQL 默认值与推荐设置

  • 默认：sync_binlog = 0（性能优先，风险较高）
  • 推荐：根据业务情况选择
  • 若要求高可靠性（金融类） → sync_binlog = 1
  • 若能容忍少量数据丢失 → sync_binlog = 100~1000

MySQL提供了一个慢日志机制

SET GLOBAL slow_query_log = ON; – 开启慢日志

SET GLOBAL long_query_time = 1; – 超过 1 秒的记录


之后所有执行时间超过 1s 的 SQL 都会被写入 slow_query.log。

SHOW VARIABLES LIKE ‘slow_query_log’;

SHOW VARIABLES LIKE ‘long_query_time’;


先执行这个命令看看有没有打开慢日志

+———————+——-+

标量子查询是 只返回一个值（一个单元格） 的子查询。
这个值可以出现在主查询的 SELECT、WHERE、HAVING 等语句中，就像一个普通字段或常量一样使用。

换句话说：
👉 标量子查询的结果相当于“先执行一个查询得到一个数字/字符串”，
然后把它当成主查询的一部分。

注意事项

1. 标量子查询必须只返回 一行一列 否则会报错

比如： SELECT (SELECT dept_name FROM dept) – 如果返回多行就报错

标量子查询和主查询的执行顺序

标量子查询（scalar subquery）可以理解为：

“在外层查询运行时，先执行里面的小查询，把结果当作一个值代入外层”。
比如 👇

执行顺序是：

1️⃣ 外层主查询 从 emp 表取出每一行员工数据。
2️⃣ 对每一行，执行内层子查询：SELECT dept_name FROM dept WHERE dept.id = emp.dept_id;

它返回部门名（一个值）。
3️⃣ 把这个值放入结果集的 deptName 列。
4️⃣ 重复步骤 2–3，直到所有员工行都处理完。

执行过程图解

如果是非相关子查询，执行更简单（性能很强，可以放心使用）

如果是“相关子查询”，每行执行一次 （性能低，慎用，除非条件固定，结果稳定）

✅ 子查询结果缓存（Subquery Cache）
当子查询的输入条件相同且结果未过期时，数据库可以复用上一次的查询结果，而无需重新计算，从而提高查询性能。
👉 因此，在子查询**条件固定、结果稳定的场景下，使用子查询通常性能较高，值得推荐**。

内连接

内连接（INNER JOIN）本质上就是笛卡尔积，只不过通常会加上连接条件。
➤ 写了条件 ⇒ 有条件的笛卡尔积（只保留匹配行）
➤ 没写条件 ⇒ 无条件的笛卡尔积（所有组合都出现）

SQL语句

SELECT 列名列表 FROM 表1 [AS] 别名1 （INNER） JOIN 表2 [AS] 别名2 ON 连接条件;

INNER关键字可省略

左连接，右连接

  • LEFT JOIN： 保留左表全部记录，右表匹配不到的部分字段用 **NULL** 填充。
  • RIGHT JOIN： 保留右表全部记录，左表匹配不到的部分字段用 **NULL** 填充。
  • ON： 指定两表的关联条件（例如 s.class_id = c.id），用于判断哪些行是匹配的。

SQL语句

SELECT 列名列表 FROM 表1 [AS] 别名1 LEFT JOIN 表2 [AS] 别名2 ON 连接条件;

SELECT 列名列表 FROM 表1 [AS] 别名1 RIGHT JOIN 表2 [AS] 别名2 ON 连接条件;

全连接

全连接（FULL JOIN / FULL OUTER JOIN）
会返回左表和右表的所有记录，

  • 匹配的行：正常合并
  • 不匹配的行：用 NULL 填充缺失部分

换句话说：

FULL JOIN = LEFT JOIN + RIGHT JOIN 的并集

SQL语句

SELECT 列名列表 FROM 表1 [AS] 别名1 FULL JOIN 表2 [AS] 别名2 ON 连接条件;

自连接

自连接（Self Join）
是指一张表自己连接自己，用于查询同一张表中存在层级关系或关联关系的数据。

本质上就是：

把同一张表当作两张表来用，通过别名区分。

SQL语句

SELECT 列名列表 FROM 表名 AS 别名1 [INNER | LEFT | RIGHT] JOIN 表名 AS 别名2 ON 连接条件;

✅ 注意：

  • 必须给表起两个不同的别名（例如 a 和 b）；
  • 否则 MySQL 无法区分同一张表的两次引用。

联合查询

联合查询（UNION）
用于将多个 **SELECT** 查询的结果纵向合并为一个结果集。

它不是“横向连接”（那是 JOIN），
而是上下拼接，把多次查询的结果叠加在一起。

SQL语句

SELECT 列名列表 FROM 表1 … UNION [ALL]

SELECT 列名列表 FROM 表2 …;

UNION 和UNION ALL的区别

UNION 会自动去重，UNION ALL 不会去重。

MySQL 的 分组查询 主要是通过 GROUP BY 子句来实现的，它将查询结果按指定的列进行分组，通常配合聚合函数（如 COUNT()、SUM()、AVG()、MAX()、MIN() 等）来对每一组数据进行计算和统计。

基本语法

SELECT column1, column2, aggregation_function(column3)

FROM table_name

GROUP BY column1, column2;


  1. **SELECT** 用于指定查询的列，包含：
  • 普通列：如 column1 和 column2，用于分组的依据。
  • 聚合函数：如 aggregation_function(column3)，对某列（例如 column3）应用聚合操作。

2. **GROUP BY**``column1, column2：按 column1 和 column2 列分组，将数据分成多个组，每个组的值在这两列上是相同的。

3. ``aggregation_function(column3)：对列 column3 的数据执行聚合操作并返回一个结果。常见的聚合函数有：

  • COUNT(column3)：返回 column3 中非 NULL 值的行数。
  • SUM(column3)：返回 column3 所有值的和。
  • AVG(column3)：返回 column3 所有值的平均值。
  • MIN(column3)：返回 column3 中的最小值。
  • MAX(column3)：返回 column3 中的最大值。
  1. 注意

在 GROUP BY 分组查询中，聚合函数中的列不一定需要出现在 <font style="color:#DF2A3F;">GROUP BY</font> 中，但如果列没有出现在 <font style="color:#DF2A3F;">GROUP BY</font> 中且没有经过聚合函数处理，它不能直接在 <font style="color:#DF2A3F;">SELECT</font> 中查询，否则查询会报错。

例子

假设有一个 employees 表，包含 department_id（部门ID）、salary（薪资）、hire_date（入职日期）等列。

按部门分组，计算每个部门的总薪资

SELECT department_id, SUM(salary) AS total_salary

FROM employees

GROUP BY department_id;

  • 这里，department_id 是分组依据，SUM(salary) 计算每个部门的总薪资。
  • salary 并没有出现在 GROUP BY 中，但它通过 SUM() 聚合函数来处理。

按部门分组，计算每个部门的最高薪资和入职最早的员工

SELECT department_id, MAX(salary) AS highest_salary, MIN(hire_date) AS earliest_hire_date

FROM employees

GROUP BY department_id;

  • department_id 是分组依据，MAX(salary) 和 MIN(hire_date) 分别用于计算每个部门的最高薪资和最早入职的日期。
  • salary 和 hire_date 并没有出现在 GROUP BY 中，但它们通过聚合函数来处理。

错误示例（如果你没有使用聚合函数处理其他列）

SELECT department_id, hire_date

FROM employees

GROUP BY department_id;

这个查询会报错，因为 hire_date 没有使用聚合函数，且它也没有出现在 GROUP BY 中。在这种情况下，数据库不能确定如何处理 hire_date，因为每个部门可能有多个不同的入职日期。

HAVING 子句

HAVING 子句的作用

  • HAVING 用于对 分组后的数据 进行过滤。它类似于 WHERE 子句，但不同的是，WHERE 用于过滤 原始数据，而 HAVING 用于过滤 已经分组 后的结果。
  • **HAVING** 常与聚合函数一起使用，用于根据聚合结果进行条件判断。

HAVING 子句的语法

SELECT column1, column2, aggregation_function(column3)

FROM table_name

GROUP BY column1, column2

HAVING condition;


  • **HAVING** 子句在分组之后应用，用来过滤聚合结果。
  • HAVING 后面可以跟聚合函数的条件（例如 COUNT()、SUM()、AVG() 等）。
  • WHERE 不能用于过滤聚合后的结果，必须先使用 GROUP BY 进行分组，再使用 HAVING 来过滤。

HAVING 和 WHERE 的区别：

  • WHERE 用于过滤 原始数据，不能与聚合函数一起使用。
  • HAVING 用于过滤 分组后的数据，可以与聚合函数一起使用。

分组查询数据链路

假设我们有以下 employees 表：

department_id employee_id salary
1 101 5000
1 102 6000
2 103 7000
2 104 8000
3 105 9000
3 106 10000

我们要执行一个按 department_id 分组的查询，来计算每个部门的 总薪资（SUM(salary)）和 员工人数（COUNT(employee_id)），并筛选出薪资总和大于 15000 的部门。

SELECT department_id, SUM(salary) AS total_salary, COUNT(employee_id) AS employee_count

FROM employees

GROUP BY department_id

HAVING SUM(salary) > 15000;


1. 原始数据：

我们从表 employees 中根据where查询出原始数据

2. 分组操作：

根据 department_id 列将数据分成三个组。

  • 组 1: department_id = 1
    • (101, 5000), (102, 6000)
  • 组 2: department_id = 2
    • (103, 7000), (104, 8000)
  • 组 3: department_id = 3
    • (105, 9000), (106, 10000)

3. 应用聚合函数：

对每个分组应用聚合函数来计算 总薪资（SUM(salary)）和 员工数（COUNT(employee_id)）。

  • 组 1:
    • 总薪资： 5000 + 6000 = 11000
    • 员工数： COUNT(2)
  • 组 2:
    • 总薪资： 7000 + 8000 = 15000
    • 员工数： COUNT(2)
  • 组 3:
    • 总薪资： 9000 + 10000 = 19000
    • 员工数： COUNT(2)

4. 查询结果：

这时，每个分组的聚合结果已被计算出来。

department_id total_salary employee_count
1 11000 2
2 15000 2
3 19000 2

5. 应用 **HAVING** 子句：

**ROW_NUMBER()** 是 SQL 中的窗口函数之一，用于为结果集中的每一行分配一个唯一的序列号。这个序列号是根据你指定的排序规则（通过 ORDER BY 子句）生成的，通常用于分页、排序、去重等场景。

语法：

ROW_NUMBER() OVER (PARTITION BY partition_column ORDER BY order_column)


  • **PARTITION BY**（可选）：用于指定分组。ROW_NUMBER() 会在每个分组内重新计数。如果没有指定 PARTITION BY，则表示整个结果集为一个分组。
  • **ORDER BY**：指定序列号的排序方式。ROW_NUMBER() 会根据这个列的顺序为每行分配一个唯一的序列号。

返回值：

**ROW_NUMBER()** 返回一个整数，从 1 开始，为每一行生成一个唯一的、按指定排序的序列号。

示例：

假设有一个 employees 表，包含以下数据：

emp_no name salary
101 Alice 5000
102 Bob 6000
103 Charlie 7000
104 David 8000
105 Eva 9000

我们想要为这个结果集按 salary 排序，并为每个员工生成一个序列号。

  1. **按 **salary** 排序使用 ****ROW_NUMBER()**：

SELECT emp_no, name, salary,

   ROW_NUMBER() OVER (ORDER BY salary DESC) AS row_num

FROM employees;


结果：

emp_no name salary row_num
105 Eva 9000 1
104 David 8000 2
103 Charlie 7000 3
102 Bob 6000 4
101 Alice 5000 5

ROW_NUMBER() 按照 salary 列从高到低排序，并为每行分配一个唯一的序列号。

  1. 使用 **PARTITION BY** 分组后进行 **ROW_NUMBER()** 排序：

假设我们对 department_id 进行分组，并在每个部门内按 salary 排序，为每个部门的员工生成一个序列号。

SELECT emp_no, name, department_id, salary,

   ROW_NUMBER() OVER (PARTITION BY department_id ORDER BY salary DESC) AS row_num

FROM employees;


结果：

假设数据如下，带有 department_id：

emp_no name department_id salary
101 Alice 1 5000
102 Bob 1 6000
103 Charlie 2 7000
104 David 2 8000
105 Eva 2 9000

执行结果：

emp_no name department_id salary row_num
102 Bob 1 6000 1
101 Alice 1 5000 2
105 Eva 2 9000 1
104 David 2 8000 2
103 Charlie 2 7000 3

在每个 department_id 中，ROW_NUMBER() 从 1 开始，按 salary 降序排列

应用查询

分页查询：

ROW_NUMBER() 常用于分页查询，结合 WHERE 子句和 BETWEEN 子句来获取特定范围的行。

WITH NumberedEmployees AS (

SELECT emp_no, name, salary, 

       ROW_NUMBER() OVER (ORDER BY salary DESC) AS row_num

FROM employees

)

SELECT emp_no, name, salary

FROM NumberedEmployees

WHERE row_num BETWEEN 10 AND 20;


去重

当我们想去除某些重复行时，ROW_NUMBER() 可以帮助我们标记重复的行，并只保留特定的行。

按特定顺序排序并标记每一行的顺序

比如，当我们需要按某种规则排序并标记每行的排名时，ROW_NUMBER() 很有用。

总结

  • **ROW_NUMBER()** 是一个窗口函数，用于根据特定排序为结果集中的每一行分配一个唯一的序列号。
  • 可以通过 **PARTITION BY** 分组，并使用 **ORDER BY** 进行排序。
  • 它广泛应用于分页查询、去重、排序等场景。

索引

定义: 索引就像目录,可以帮助快速定位找到数据,同时维护索引也需要额外的空间,所以索引是一种空间换时间的方式

聚簇索引与非聚簇索引的不同

聚簇索引中数据和索引是放到一起的找到了索引也就找到了数据

非聚簇索引数据与索引分开存放,索引指向数据,所以找到索引还要再额外找一次才能找到是数据

相比于非聚簇索引,聚簇索引的有点肯定是查找是速度更快,缺点是当更新索引的时候代价更大因为要移动整个数据

为什么mysql 喜欢b+树

  1. b+树相对于b树: b+树只有叶子节点存储数据,所以对于非叶子节点每次进行磁盘io 的时候b+树总能查询更多节点,效率更高,另外b+树的叶子节点用双链表链接,很适合进行顺序查找
  2. b+树相对于二叉树: 二叉树的高度太高,磁盘io的次数更多效率十分低下
  3. b+树相对与hash: hash 在做等值查询的时候效率很快是o(1)，但是hash 不能做范围查找,只能做等值查找

索引的优点和缺点

  1. 优点:查找效率更快
  2. 缺点:缺点是维护索引需要成本,占用空间,同时删除更新索引也要进行维护

一.Explain是什么

EXPLAIN用来分析MySQL查询语句的执行计划，它不会真正的执行SQL，只会展示执行步骤，包括

  • 表的访问顺序（谁先谁后）
  • 使用的索引
  • 是否全表扫描
  • 每一步大约扫描多少行

二.EXPLAIN输出字段解释

字段 含义 示例 说明
id 查询中每个 SELECT
的执行顺序标识		 1, 2 id 越大优先级越高（先执行）
select_type 查询类型 SIMPLE, PRIMARY, SUBQUERY 表示是否是子查询、联合查询等
table 当前访问的表名 user, orders MySQL 处理的具体表或临时表
partitions 表分区信息 NULL 通常不用关注
type 连接类型（访问方式） index, range, ref, const, ALL 最关键指标之一，越靠前性能越好
possible_keys 可能用到的索引 idx_age, idx_name 优化器考虑过的索引
key 实际使用的索引 idx_age 如果为 NULL 表示未使用索引
key_len 使用索引的字节长度 5 越短越精准越好
ref 索引匹配的列或常量 const 用哪个列与索引匹配
rows 预计要扫描的行数 100 估算值（越小越好）
filtered 条件过滤比例 10 百分比，表示过滤后的行比例

三.EXPLAIN关键字段

id —— 执行顺序

  • 表示查询中每个 SELECT 的执行标识。
  • 数字越大，执行优先级越高。

例如:EXPLAIN SELECT * FROM user WHERE id IN (SELECT user_id FROM order);


输出中会有两行：

  • 子查询的 id 更大，说明子查询先执行。

type —— 访问类型（最重要)

表示MySQL如何查找表中的行， 这是 SQL 执行性能最直观的指标之一。 性能由好到查排序如下

type 值 含义 说明 举例
system 只有一行数据的系统表 特殊情况，极快 SELECT * FROM dual;
const 通过主键或唯一索引精确匹配 只返回一行，常量级别 WHERE id = 1
eq_ref 联表时主键等值匹配 多表连接中，每次都只匹配一行 user.id = order.user_id
且 user_id 为主键
ref 非唯一索引或前缀索引等值匹配 可能返回多行 WHERE name = 'Bob'
且 name 有普通索引
range 索引范围扫描 范围条件查询 WHERE age > 18
或 BETWEEN
index 全索引扫描 扫描整个索引树，而非表 没WHERE但用到索引顺序，如 ORDER BY name
ALL 全表扫描 没索引最慢 SELECT * FROM users

key —— 实际使用的索引

  • 显示优化器真正选择的索引。
  • 若为 NULL → 没有使用索引。
  • 若不为空 → 表示命中索引。

对比：

possible_keys: idx_name, idx_age
key: idx_age


说明优化器考虑了两个索引，但最终选择了 idx_age。

rows —— 预计扫描的行数

  • 表示 MySQL 估计要扫描的记录数量。
  • 越小越好。
  • MySQL 会根据表的统计信息估算。

Extra —— 附加信息（隐藏性能陷阱）

它描述了 MySQL 执行查询时的额外操作，也就是说：除了索引访问以外，MySQL 在执行过程中还“多做了什么”。

常见的Extra值详解

Extra 含义 性能影响 说明/场景
Using index 覆盖索引查询 ✅ 优化很好 数据只从索引中获取，不访问表数据（不回表）
Using where 使用 WHERE 条件过滤 ⚠️ 正常 表示 MySQL 还要额外判断 WHERE 条件
Using temporary 用了临时表 ❌ 性能差 出现在 GROUP BY
、DISTINCT
、UNION
Using filesort 用了文件排序 ❌ 性能差 出现在 ORDER BY
，说明没用上索引排序

Using join buffer 联表时用了连接缓冲区 ⚠️ 需优化 表示没用上索引连接
Using MRR 多范围读优化 ✅ 改善随机 I/O
Using where; Using index 同时过滤+覆盖索引 ✅ 很好 用索引过滤并避免回表
NULL 没有额外操作 ✅ 最理想

Extra中的using where 含义

extra中using where 并不是说SQL中出现了WHERE语句而是表示MySQL再执行时需要额外过滤

比如:

EXPLAIN select * from users where id=2


这个sql就不需要执行额外过滤，它检索到id=2的数据就好了，所以Extra 就为null

EXPLAIN select * from users where id>2


这个sql当它检索到第一个id>2的数据后还需要进行额外的过滤继续查找，所以Extra就是using where

_注意 : 即使 SQL 语句中没有写 _WHERE_，但如果 MySQL 必须检查行是否满足某个条件（比如 join 条件、HAVING、子查询匹配等），它仍然会显示 _Using where_。 _

四.EXPLAIN ANALYZE与EXPLAIN区别

项目 EXPLAIN EXPLAIN ANALYZE
是否执行 SQL ❌ 不执行，只显示计划 ✅ 真正执行并统计数据
作用 预测执行计划（优化器预估） 实际执行报告（真实耗时与行数）
输出内容 访问方式、索引、预估行数等 每步的实际耗时、扫描行数、回表次数等
用途 快速分析索引使用情况 精准定位性能瓶颈（排序/回表/连接）

一句话记忆：

EXPLAIN 看计划，EXPLAIN ANALYZE 看实战。

五.根据EXPLAIN ANALYZE分析过程

假设执行SQL语句

EXPLAIN ANALYZE SELECT * FROM users WHERE age > 18 ORDER BY create_time DESC;

输出示例

-> Sort: users.create_time desc (cost=100 rows=100) (actual time=0.1..2.3 rows=95 loops=1)

-> Filter: (users.age > 18)  (cost=80 rows=100) (actual time=0.05..1.9 rows=95 loops=1)

    -> Table scan on users  (cost=40 rows=1000) (actual time=0.03..1.7 rows=950 loops=1)

箭头->

表示执行顺序，最下层先执行。

  • 越靠右的行，执行得越“底层”；
  • 越靠左的行，执行得越“外层”。

操作类型

如：

  • Table scan：全表扫描
  • Index range scan：范围索引扫描
  • Nested loop join：嵌套循环连接
  • Sort / Filter：排序、过滤阶段

cost=xxx rows=xxx

这是优化器预估值（plan 阶段产生的预测,预估值可能不准）。

  • cost：优化器计算的相对代价（非真实时间）
  • rows：预估扫描的行数

👉 用于比较预测 vs 实际是否偏差大。、

actual time=起始..结束

这是实测耗时：

  • 单位：毫秒
  • 第一个数是“开始时间”，第二个是“结束时间”
    → 0.1..2.3 表示从开始到结束共执行 2.2ms

rows=xxx

这是真实返回行数（可能和预估不同）。
如果差距太大，说明优化器估算不准，可能索引统计信息老化。

loops=xxx

表示该操作执行了多少次。
典型在 JOIN 场景下很重要，比如：

-> Nested loop inner join (actual time=0.02..5.00 rows=100 loops=100)

说明内层循环被执行了 100 次（可能是性能瓶颈）。

六.为什么建议先使用EXPLAIN再使用EXPLAIN ANALYZE

两者定位不同

命令 性质 是否执行SQL 使用场景
EXPLAIN 预估计划 ❌ 不执行 安全查看执行路径、索引使用
EXPLAIN ANALYZE 实测报告 ✅ 真执行SQL 精确测慢点、验证优化结果

为什么要“先 EXPLAIN 再 ANALYZE”

EXPLAIN 是安全预览

  • 它**不会真的执行 SQL**，不会修改数据、不会扫描整表。
  • 如果 SQL 写错或特别慢（比如误写 SELECT * FROM big_table），
    用 EXPLAIN ANALYZE 会直接把数据库卡住。

EXPLAIN ANALYZE 开销更大

  • 它**真的执行 SQL**，要扫描数据、做排序、JOIN 等操作。
  • 对大表来说，几秒、几十秒都可能。
  • 如果 SQL 写错或是更新语句，可能直接改动生产数据 ⚠️

b树

关于m 阶b树的节点 最多有m个分支 最少有m的一半向上取整个分支(保证b树某个节点不会太稀疏)

b+ 树

b+树的特点就是叶子节点用一个链表连接起来 ,并且每个非叶子节点的值都是其下层节点的最大值

b+树和b+的区别

可以很明显的看到b+树叶子节点包含着所有的数据的索引和指针,而b树每层节点都包含数据的索引和指针

所以b+树很适合顺序查找,b 树如果要进行顺序查找的话需要中序遍历,而b+树只需要定位到首元素然后通过链表进行顺序查找即可

b+ 树也很适合进行范围查找 比如说查找10-50内的元素,b+树先查找大于元素10的元素然后按链表顺序进行查找即可
