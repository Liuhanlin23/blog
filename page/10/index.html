堆内存中主要分为两个区域

新生区（新生代）

  • 专门用来存放 刚创建出来的新对象。
  • 垃圾回收频繁（称为 Minor GC）。
  • 采用 复制算法（Copying GC） 来回收。

  • 大量新对象进入 Eden；
  • 当 Eden 满时，触发 Minor GC；
  • 仍然存活的对象会被复制到一个 幸存区；
  • 多次 Minor GC 后还存活的对象，会被移动到老年代（称为 **晋升 **）。

养老区（老年代）

  • 存放“寿命较长”的对象或者大对象。
  • 大对象直接进入老年代的行为是由虚拟机动态决定的，它与具体使用的垃圾回收器和相关参数有关。大对象直接进入老年代是一种优化策略，旨在避免将大对象放入新生代，从而减少新生代的垃圾回收频率和成本。
  • 当对象在新生代经历多次 GC 仍未被回收时，会被晋升到老年代。

引用计数法

引用计数法（Reference Counting）是一种 简单直接的垃圾回收策略：

每个对象维护一个“引用计数器”，
每当有一个引用指向它时，计数 +1；

当计数为 0 时，说明该对象“不可再被访问”，可以被回收。

最大问题：无法处理循环引用

这是引用计数法被 JVM 淘汰的关键原因。

如果a和b互相引用，此时即使我们把外界对a和b的引用都置null，它们也不会被gc因为它们相互引用，引用计数不为0

所以jvm不采用引用计数法因为循环引用在java中太常见了比如说双向链表

标记清除算法

标记清除算法就是分两步

  • 标记活着的对象
  • 对死亡的对象进行清除

标记整理算法

标记整理算法就是优化标记清除算法的内存碎片问题，分三步

  • 标记活着的对象
  • 把所有活着的对象往一端移动
  • 清理边界外的空间（死亡对象）

一.什么是full gc

Full GC（Full Garbage Collection）是一次针对整个堆（新生代 + 老年代）以及方法区（JDK8 以后是元空间）的完全垃圾回收。

元空间在直接内存中为什么还会被gc呢？

元空间在直接内存中，直接内存不受jvm 管理不会发生gc，那为什么full gc 还会回收它呢？

直接内存本身不会被gc扫描，但JVM会在GC过程中，通过类卸载机制间接释放元空间

如何理解类卸载

首先类加载器lassLoader 本质上就是一个普通的 Java 对象，只不过它的职责是“加载类”。 它也在 Java 堆（Heap） 里分配内存，也会被 GC 管理。 JVM 会把每个类与它的 ClassLoader 绑定起来。只要你还在用这个 ClassLoader 加载的类、对象、静态变量……它就活着。

所以类卸载的步骤如下：

  1. full gc 触发（minor gc 不会触发类卸载）
  2. 发现某个类加载器不可达
  3. 卸载该ClassLoader
  4. 卸载它所加载的所有class（元空间中的），注意元数据空间中类卸载释放内存一定不是通过gc完成的因为直接内存中没有gc，是通过元空间管理器完成的
  5. 释放元空间内存

二.什么时候会发生full gc

触发原因 说明 示例 / 场景
老年代空间不足 新生代对象晋升时，老年代放不下，触发 Full GC 长生命周期对象太多、缓存未及时清理
调用 System.gc() 显式调用请求一次完整 GC（通常会触发） 代码中手动调用 System.gc()
或框架触发
元空间（Metaspace）不足 类加载太多导致元空间耗尽 大量动态代理类、频繁热加载 JSP 或插件

三.常见导致频繁full gc的场景

  1. 内存泄露

应用长时间运行后，老年代内存占用持续增长，即使经过多次Full GC也无法有效释放空间，最终可能导致OOM

  1. 高并发场景

在高并发、高吞吐量的服务中，大量对象在短时间内被创建并快速晋升到老年代，导致老年代空间不足，触发频繁Full GC。

  1. 大对象分配场景

场景描述：应用程序中频繁创建大对象或大数组，这些对象直接分配在老年代，导致老年代空间不足，触发Full GC。

  1. 内存碎片化场景

场景描述：老年代中虽然有足够的空闲空间总量，但由于空间碎片化，无法找到足够大的连续空间来分配对象，导致Full GC。

  1. 元空间溢出

在JDK 8及以上版本中，Metaspace空间不足导致频繁Full GC，最终可能引发OutOfMemoryError: Metaspace。

  1. jvm 参数配置不合理

由于JVM参数配置不合理，导致GC策略不适合当前应用特性，引发频繁Full GC。

  1. 频繁调用gc方法

应用代码中直接调用System.gc()或Runtime.getRuntime().gc()方法，或者RMI等机制定期触发Full GC。

四.full gc 的排查方案

开启分析gc 日志

GC日志是排查Full GC问题的最基本和最重要的工具。通过分析GC日志，可以了解GC的频率、持续时间、内存使用情况等关键信息。

监控jvm 内存使用情况

使用如VisualVm工具+Visual GC插件实时监控JVM内存使用情况，注意观察

  • Eden 区
  • Survivor 区
  • Old Gen

分析jvm 堆中的对象实例

使用jmap 命令把jvm 堆里的所有对象实例完整导出到一个文件中，进行分析。

当执行jamp 命令后，jvm会

  • 暂停所有线程（安全点，Safepoint），保证堆在一个稳定状态；会造成业务的停滞
  • 遍历整个堆（年轻代 + 老年代 + Survivor 区）；
  • 把所有对象的：
  • 类名（Class）
  • 对象大小
  • 对象引用关系（谁引用了谁）
  • 字段值（基本类型值、对象引用）
  • 类加载器等元信息
    全部写入 .hprof 文件。

新switch

使用箭头->简化写法

旧写法

switch (x) {

case 1:

    foo();

    break;

case 2:

    bar();

    break;

}


新写法

switch (x) {

case 1 -> foo();

case 2 -> bar();

}


  • 无需 break
  • 不会穿透
  • 可读性高

switch 可以返回值

以前的switch 只能最为语句，不能返回值

现在可以

String result = switch (day) {

case 1 -> "Mon";

case 2 -> "Tue";

default -> "Other";

};


switch 本身是一个“表达式”了。

什么是caffeine？

Caffeine 是基于 Java 8 的高性能本地缓存库，常与 Redis 搭配组成多级缓存架构。它在 HashMap 和 ConcurrentHashMap 的基础上进行了增强，封装成一个智能缓存框架，额外提供了 线程安全、过期控制、容量限制、淘汰策略、自动加载、统计监控 等高级功能。

使用

创建Cache

在 Caffeine 中，Cache<K, V> 就是一个 线程安全的本地内存缓存容器，它底层是一个并发优化过的 ConcurrentHashMap + 一套 “智能过期与淘汰策略”。

配置缓存策略

方法 功能 说明
.maximumSize(10_000) 设置最大缓存条目数 超出会自动淘汰旧数据（基于 LRU + W-TinyLFU）
.expireAfterAccess(5, TimeUnit.MINUTES) 访问后 5 分钟过期 每次读写都会刷新这个计时器
.recordStats() 开启统计功能 之后可以通过 cache.stats()
查看命中率、加载耗时

常用方法

方法 说明 示例
put(K key, V value) 手动放入缓存 cache.put(1L, user)
getIfPresent(K key) 获取缓存，不存在返回 null User u = cache.getIfPresent(1L)
get(K key, Function<K,V> mappingFunction) 不存在则加载新值 cache.get(id, this::loadFromDB)
invalidate(K key) 删除指定 key cache.invalidate(1L)
invalidateAll() 清空缓存 cache.invalidateAll()
asMap() 转为 ConcurrentMap cache.asMap().forEach(...)
estimatedSize() 估算当前大小 System.out.println(cache.estimatedSize())
cleanUp() 手动触发清理（通常不用） cache.cleanUp()
stats() 查看命中率统计（需 recordStats） cache.stats().hitRate()

一.对称加密

  1. 特点：加密和解密使用同一把“密钥”
  2. 优点：速度快（加密解密效率高）
    缺点： 密钥需要共享给对方->容易被窃取；一旦泄露，通信内容可被任意加解密。

二.非对称加密

  1. 特点：一对密钥——公钥 + 私钥
  • 公钥：可以公开，用来加密
  • 私钥：自己保留，用来解密

举例：

（服务器）公开我的公钥🔓，
（浏览器）用它加密消息发给我，
只有服务器（持有私钥🔐）能解密。同时服务器公开的是公钥，即使泄露也只能对数据进行加密而不能解密

  1. 优点：安全（私钥不需要传输）
    缺点：速度慢（计算复杂）
