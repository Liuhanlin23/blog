HTTPS

分类于 网络

一.HTTP与HTTPS的区别

对比项 HTTP HTTPS
全称 HyperText Transfer Protocol HyperText Transfer Protocol Secure
端口号 80 443
是否加密 ❌ 明文传输，容易被窃听或篡改 ✅ 加密传输（TLS/SSL）保证机密性和完整性
安全机制 无 使用 SSL/TLS 协议
证书 不需要 需要 数字证书（CA颁发） 验证服务器身份
性能 快一些（无加解密） 稍慢一点（有加解密开销）但可用硬件/协议优化（如 HTTP/2）
数据完整性 无校验机制 有完整性校验（防止被篡改）
典型用途 一般网站、内部服务 登录、支付、银行、电商、隐私相关业务

二.HTTPS的加密过程

https只是在http协议和tcp层之间加了一个TLS(原SSL)的安全传输协议

TLS工作原理

TLS 的核心思想：结合对称加密与非对称加密，兼顾安全与效率。

  • 对称加密：同一密钥加解密，效率高但密钥传输不安全。
  • 非对称加密：公钥加密、私钥解密，安全但效率低。
    → TLS 用“非对称加密”安全地传输“对称密钥”，再用“对称加密”进行高速通信。

基本流程

  1. 服务器 生成一对非对称密钥（公钥 + 私钥）。
  2. 公钥 通过数字证书下发给客户端。
  3. 客户端 生成随机的对称密钥，用服务器公钥加密后发送给服务器。
  4. 服务器 用私钥解密，得到相同的对称密钥。
  5. 之后双方使用该对称密钥进行加密通信。

存在的问题

若公匙直接明文传输公钥，可能被中间人伪造。
因此 TLS 通过 数字证书（CA 签名） 验证公钥的真实性，防止中间人攻击。

为什么明文传输公匙会有中间人攻击

假设攻击者在中间拦截了你和服务器之间的通信：

你 ←→ 攻击者 ←→ 服务器

他可以：

  1. 拦截服务器发来的公钥；
  2. 换成自己生成的一对公钥/私钥，发给你；
  3. 你以为是服务器的公钥，实际是攻击者的；
  4. 之后你发的加密数据，攻击者能用自己的私钥解密；
  5. 他再用服务器公钥重新加密转发——你和服务器都没发现被劫持。

这就是典型的中间人攻击

CA证书

CA 证书是什么

CA (Certificate Authority) 是数字证书，用来证明网站的身份真实性，防止公钥在传输中被篡改或伪造。

工作原理

证书签发过程

  1. 网站生成一对密钥（公钥 + 私钥）。
  2. 将公钥 + 域名等信息打包成 CSR 申请文件 ，提交给 CA。
  3. CA 验证网站身份（确认确实是 bank.com）。
  4. CA 使用 CA 私钥 对证书内容的 hash 摘要 进行加密，生成 数字签名。
    ✅ 生成的证书包括：网站公钥、域名信息、CA 签名。

注意： 签名只是对 hash 摘要加密，而不是加密整个证书。

浏览器的验证过程

  1. 服务器把证书发给浏览器。
  2. 浏览器提取证书中的：
    • 公钥信息
    • CA 的数字签名
  3. 浏览器使用 CA 公钥 (系统内置) 解密签名，得到摘要 A。
  4. 再对证书明文部分重新做 hash，得到摘要 B。
  5. 比较 A 和 B：
    • 相同→ 证书未被修改，是合法的
    • 不同→ 证书被篡改，拒绝连接。

Spring 的代理对象

分类于 SpringBoot

什么是代理对象

代理对象就是一个“替身对象”，它包裹了你的原始对象（目标对象），拦截你对目标对象的调用，在前后插入一些额外逻辑。

在 Java 中，这种代理对象通常由两种机制实现：

  1. JDK 动态代理（JDK Proxy）
  2. CGLIB 字节码代理

Spring 会根据 Bean 的情况自动选择其中一种。

为什么 Spring 要创建代理对象？

为了实现AOP（面向切面编程）、事务控制、安全检查、缓存注解等等。这些特性都需要在方法执行前后“插入逻辑”，而不修改原代码。

例如

在 Spring 中，@Transactional 依靠 AOP 代理机制 实现。

当我们调用：userService.saveUser();时，实际上并不是调用的原始的UserService，而是Spring 创建的代理对象，这个代理对象会在方法执行的前后自动插入事务逻辑

动态代理和静态代理的区别

  1. 静态代理是指：代理类在编译时就已经存在，由程序员手写或编译生成。也就是说，你要先写一个接口、一个真实类（被代理类），再写一个代理类，在里面调用真实类的方法。
  2. 动态代理是指：代理类在运行时动态生成，而不是编译时写死的。

Spring 三级缓存

分类于 SpringBoot

什么是Spring 三级缓存

Spring 的三级缓存是用来提前暴露未完全创建的 Bean 对象，从而解决单例 Bean 的循环依赖问题的机制。

例如

A 依赖 B，B 又依赖 A。
👉 如果直接反射 new，就会死循环。

所以 Spring 必须想办法“先创建一半”，把引用提前暴露出去。这就是三级缓存登场的地方。

三级缓存结构 （核心类 DefaultSingletonBeanRegistry）

Spring 在这个类里维护了三个Map

缓存层级 名称 作用 何时存放 何时清除
一级缓存 singletonObjects 已经完全初始化好的单例 Bean（可直接使用） Bean 创建完成后放入 容器关闭时清空
二级缓存 earlySingletonObjects 提前曝光的 Bean 实例（已 new 但未完全初始化） 遇到循环依赖时提前暴露 创建完成后清空
三级缓存 singletonFactories 可以创建“早期对象”的工厂（ObjectFactory） 实例化后、属性注入前放入 当对象被取出或转移到二级缓存后清空

创建对象过程

1️⃣ 创建 A

  • 通过反射调用构造器实例化出 A（实例化阶段完成，但还未注入属性）。
  • 注册 A 的早期对象工厂：
  • singletonFactories[“A”] = () -> getEarlyBeanReference(“A”, A实例）
  • （这一步只是把一个“可以返回 A 早期引用的工厂函数”放入三级缓存）
  • 进入注入A属性阶段，发现依赖B->执行getBean(“B”) 去创建B

2️⃣ 创建 B（因为 A 依赖 B）

  • 通过反射调用构造器实例化出 B（实例化阶段完成）。
  • 注册 B 的早期对象工厂：
  • singletonFactories[“B”] = () -> getEarlyBeanReference(“B”, B实例)
  • 进入注入B属性阶段，发现依赖A->执行getBean(“A”)

3️⃣ 在创建 B 的过程中注入 A

调用 getBean("A") 时，Spring 依次查找：

  • 一级缓存（singletonObjects）：❌ 没有
  • 二级缓存（earlySingletonObjects）：❌ 没有
  • 三级缓存（singletonFactories）：✅ 有 “A 的工厂”
  • 调用该工厂： 生成 A 的早期引用（可能是原始 A，也可能是 AOP 代理对象）。 将这个早期 A 放入二级缓存：
  • 使用这个早期 A 给 B 注入依赖。

4️⃣ B 创建完成

  • B 的依赖注入与初始化都完成 → 放入一级缓存：
  • 清理 B 的二级、三级缓存记录。

5️⃣ 回到 A，完成剩余步骤

  • 继续给 A 注入 B（此时 B 已在一级缓存中，可以直接注入）。
  • A 完成依赖注入与初始化 → 放入一级缓存：
  • 清理 A 的二级、三级缓存记录。

为什么是三级缓存而不是二级缓存

如果只有二级缓存，会出现什么问题

假设我们只有：

流程是这样的：

  1. A 实例化 → 放入二级缓存
  2. B 创建 → 需要 A → 从二级缓存拿 A → 成功
  3. 然后 A 完成初始化（此时 AOP 代理可能介入）→ A 最终变成代理对象。

问题就来了：

此时 B 已经拿到了“原始的 A 对，而不是代理后的对象！

也就是说：

  • A 最终在容器中是 AProxy
  • B 注入的是 A 原对象
  • 这会让 AOP 失效（B 调用 A 的方法绕过了代理）

💥 二级缓存无法解决 “A 被代理后，B 拿到的 A 不是最终版本” 的问题。

spring Bean 声明周期

分类于 SpringBoot

一.简要概括

好的，下面是对 Spring Bean 生命周期四个阶段的一句话概括：

  1. 实例化（Instantiation）：Spring 通过反射创建 Bean 对象的实例。
  2. 属性赋值（Populate Properties）：Spring 将配置文件或注解中定义的依赖注入到 Bean 的属性中。（三级缓存）
  3. 初始化（Initialization）：Spring 调用 Bean 的初始化方法（如 @PostConstruct、afterPropertiesSet() 或自定义 init-method）进行初始化逻辑。
  4. 销毁（Destruction）：当容器关闭时，Spring 调用销毁方法（如 @PreDestroy、destroy() 或自定义 destroy-method）释放资源。

二.初始化

初始化阶段的目标：让 Bean 完全准备好可以使用。 经过第一二步 反射出来的对象现在还只是“一个有属性的普通对象”；Spring 要在初始化阶段执行一系列“钩子（回调）”，让 Bean 真正“活起来”。

初始化阶段执行顺序

1️⃣ 实例化（new 对象）

2️⃣ 属性注入（依赖注入）

3️⃣ 调用 Aware 接口（如 BeanNameAware）

4️⃣ BeanPostProcessor 前置处理

➡️ 5️⃣ @PostConstruct（执行初始化逻辑）

6️⃣ afterPropertiesSet() —— 来自 InitializingBean 接口

7️⃣ init-method() —— 在 XML / @Bean 中指定

8️⃣ BeanPostProcessor 后置处理（常用于 AOP 代理）


Bean的初始化方法

@PostConstruct（重点 ⭐）

  • 作用：在 Bean 完成依赖注入后、但在使用前自动执行一次初始化逻辑。
  • 执行时机：依赖注入完成 → 初始化前（在 AOP 代理之前）。

@PostConstructor 方法要求

  • 返回值类型：void
  • 无参数（不能带任何参数）
  • 不能抛出受检异常（checked exception）

afterPropertiesSet 与 init-method（了解）

方法 定义方式 调用时机 场景

接口		 @PostConstruct 之后 用于框架级初始化
init-method() 在 XML 或 @Bean 指定 afterPropertiesSet 之后 用于配置文件初始化

三者都能定义初始化逻辑，执行顺序为：
@PostConstruct → afterPropertiesSet → init-method

运行时发生了什么？

当 Spring 创建完 Bean 实例并注入属性后，会在初始化前执行👇：

invokeAwareMethods(beanName, bean);


Spring 会判断当前 Bean 是否实现了这些接口：

if (bean instanceof BeanNameAware) {

((BeanNameAware) bean).setBeanName(beanName);

}

if (bean instanceof BeanFactoryAware) {

((BeanFactoryAware) bean).setBeanFactory(this);

}

if (bean instanceof ApplicationContextAware) {

((ApplicationContextAware) bean).setApplicationContext(context);

}


也就是说：
Spring 主动调用这些 setXXX() 方法，把容器对象传进去。这样 Bean 就能拿到容器级资源了。

AOP

分类于 SpringBoot

** AOP（面向切面编程）是一种将公共逻辑抽取并封装为切面的编程思想，它能够在需要的地方自动切入，从而实现代码复用，同时保持业务代码的简洁与清晰。**

  1. 优点：
  • 代码解耦：公共逻辑与业务逻辑分离，业务代码更干净。
  • 复用性强：一个切面可以应用到多个方法，减少重复代码。
  • 灵活扩展：新增公共逻辑时，只需修改切面，无需改动业务代码。
  1. 缺点：
  • 调试复杂：切面逻辑可能影响多个方法，错误时需要追踪切面代码。
  • 性能开销：AOP通过动态代理实现，大量切面可能增加方法调用耗时（但日常业务中可忽略）。

核心概念

  1. 切面(Aspect)： 编写执行逻辑,里面定义拦哪些点（ @Pointcut ）,做哪些事（@Advice）
  2. 切入点：要切入的方法
  3. 通知: 在切入点执行的逻辑

类内this调用自己失效解决方案

类内调用失效的本质原因是：AOP 依赖代理生效，而 this.xxx() 是直接调用目标对象本身，不经过代理，自然无法触发 AOP。

可以有以下两种解决方案

将方法抽离为bean

打开exposeProxy

在启动类或任何一个配置类上加： @EnableAspectJAutoProxy(exposeProxy = true)

在类内调用this.xxx方法时先获得这个类的proxy 代理类

内部实现

AopContext.currentProxy() 本质上就是从当前线程的 ThreadLocal 中读取“当前正在执行的代理对象”。
Spring 会在进入代理方法时将该代理写入 ThreadLocal，方法执行完再恢复，因此不同线程互不影响。
