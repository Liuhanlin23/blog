HTTP2 相对于HTTP1 的五大核心优势

序号 优势点 对应解决了 HTTP/1.1 的什么问题 一句话理解
1️⃣** 多路复用 (Multiplexing)** HTTP/1.1 同一连接只能排队处理一个请求，容易“队头阻塞” ✅ 一个连接可同时传多个请求和响应，互不干扰 让多个请求在同一连接中并行传输，不再排队等待。
2️⃣** 二进制分帧 (Binary Framing)** HTTP/1.1 使用纯文本格式，解析和传输效率低 ✅ 将消息拆成二进制帧，提高解析速度和可靠性 把文本协议改为二进制帧结构，传输更快更稳。
3️⃣** 头部压缩 (HPACK)** 每次请求都要重复传很长的 Header（Cookie、User-Agent 等） ✅ 使用压缩算法和索引表，大幅减少头部开销 通过压缩与索引减少重复头部传输，节省带宽。
4️⃣** 服务器推送 (Server Push)** 客户端只能被动请求资源 ✅ 服务器可主动推送资源（如 CSS、JS），减少延迟 服务器可提前推送依赖资源，但已被 HTTP/3 废弃。
5️⃣** 单连接复用（减少 TCP 连接数）** HTTP/1.1 需要多个 TCP 连接并行，资源浪费 ✅ 一个连接即可传所有请求，握手少、延迟低 一个连接跑完所有请求，握手更少，传输更快。

HTTP2的多路复用解决了HTTP1.1的什么问题

在 HTTP/1.1 中：
● 一个 TCP 连接一次只能处理一个请求；
● 如果前一个请求没返回，下一个请求就得等；
● 对同一个域名，浏览器最多同时开 6 个连接来提升并发；
● 但开启过多连接会浪费资源、占用带宽，毕竟 TCP 连接的建立与销毁（握手、挥手）都十分昂贵。

HTTP2多路复用（Multiplexing）：
在 一个 TCP 连接 里，同时并行地发送多个 HTTP 请求和响应，
每个请求都有独立的 Stream ID（流编号），互不干扰。

也就是说：

HTTP/2 不再为每个请求都开新连接，而是让所有请求共享一个连接。
数据在同一个连接中被拆分成一帧一帧（Frame）交错发送，
浏览器和服务器根据 Stream ID 把属于同一个请求的帧重新组装起来。

解决的问题

HTTP/1.1 的问题 HTTP/2 的改进
每个请求必须独占一个 TCP 连接（或排队等待） ✅ 所有请求共享一个连接
串行传输 → 队头阻塞严重 ✅ 并行传输 → 不互相阻塞
建立多个连接浪费资源 ✅ 一个连接复用 → 握手少、资源省
浏览器需要“域名分片”才能提高并发 ✅ 不再需要域名分片

HTTP2的多路复用只会为每个域名建立一个连接么？

“HTTP/2 一般会为同一个域名维持一个长连接，所有请求在该连接内通过多路复用并行传输；
不同域名、协议或端口的请求仍会建立各自的连接。”

什么是串行传输->队头阻塞

什么是串行输出

在 HTTP/1.1 时代，一个 TCP 连接上：

一次只能处理一个请求。

也就是说：

  1. 浏览器发一个请求（例如请求 index.html）；
  2. 必须等服务器返回完整响应后，
  3. 才能在同一个连接上发送下一个请求。

为什么会队头阻塞

“队头阻塞”就是说：

当队列最前面的请求（队头）没有完成时，
后面的请求都被迫等待，哪怕它们其实可以先返回。

比如：

  • 浏览器开了一个连接；
  • 第一个请求 /index.html 卡住（服务器慢、网络抖动）；
  • 后面的 /style.css、/app.js 都得等；
  • 整个页面加载被拖慢。

就像排队买奶茶：第一个人犹豫半天不点单，后面的人都动不了。

对于HTTP1.1的对头阻塞浏览器的临时补救策略

为了减轻这种队头阻塞，浏览器在 HTTP/1.1 里做了“并发连接限制”：

  • 对同一域名最多开 6 个 TCP 连接；
  • 每个连接上依然是串行的；
  • 这样多个请求可以分配到不同连接中同时跑，但依然浪费资源。、
HTTP2为什么可以解决

HTTP/2 把这种「一根吸管排队」的模式，变成了「一根多通道吸管」：

  • 同一个连接里能并行发多个请求；
  • 每个请求分配一个独立的流（Stream ID）；
  • 互不影响，从根本上解决了应用层的队头阻塞。

HTTP2多路复用的实现原理

首先HTTP1.1的消息格式是纯文本，太过简单

这一串要完整传完、解析完，服务器才能处理。
所以——一个连接里不能同时传两个请求。

HTTP/2 的核心改进是：

在 TCP 之上，引入一层 二进制分帧层（Binary Framing Layer）。

HTTP/2 的实现核心：二进制分帧 + 流（Stream）

HTTP/2 把通信过程切分成很多“帧（Frame）”：

  • 每个帧是最小的传输单位；
  • 每个帧都包含一个 Stream ID（流 ID）；
  • 每个流代表一个独立的 HTTP 请求/响应，一个流对应多个帧。

例如， 浏览器要发三个请求：

Stream 1 -> index.html

Stream 3 -> style.css

Stream 5 -> app.js

** HTTP/2 会把它们的数据都拆成帧： A1, B1, C1, A2, B2, C2 … 并在同一个 TCP 连接中交错发送。 **

HTTP2提供的其他功能

HTTP/2 还提供了：

  • Stream 优先级（Priority）：告诉服务端哪个请求更重要；
  • 流量控制（Flow Control）：防止某个流独占带宽（用 WINDOW_UPDATE 帧调节）。

二级分帧

为什么要引入二进制分帧

在 HTTP/1.1 中：

  • 所有数据（包括头、体）都是纯文本格式；
  • 解析要靠换行符、空格等标识；
  • 对计算机来说效率很低，也不容易并行。

HTTP/2 想要实现「多路复用」，就必须让通信内容能被精确地分割、标识、交错、重组。所以，它放弃文本格式，引入了一个新的层 ——👉 二进制分帧层（Binary Framing Layer）。

对比项 HTTP HTTPS
全称 HyperText Transfer Protocol HyperText Transfer Protocol Secure
端口号 80 443
是否加密 ❌ 明文传输，容易被窃听或篡改 ✅ 加密传输（TLS/SSL）保证机密性和完整性
安全机制 无 使用 SSL/TLS 协议
证书 不需要 需要 数字证书（CA颁发） 验证服务器身份
性能 快一些（无加解密） 稍慢一点（有加解密开销）但可用硬件/协议优化（如 HTTP/2）
数据完整性 无校验机制 有完整性校验（防止被篡改）
典型用途 一般网站、内部服务 登录、支付、银行、电商、隐私相关业务

二.HTTPS的加密过程

https只是在http协议和tcp层之间加了一个TLS(原SSL)的安全传输协议

TLS工作原理

TLS 的核心思想：结合对称加密与非对称加密，兼顾安全与效率。

  • 对称加密：同一密钥加解密，效率高但密钥传输不安全。
  • 非对称加密：公钥加密、私钥解密，安全但效率低。
    → TLS 用“非对称加密”安全地传输“对称密钥”，再用“对称加密”进行高速通信。

基本流程

  1. 服务器 生成一对非对称密钥（公钥 + 私钥）。
  2. 公钥 通过数字证书下发给客户端。
  3. 客户端 生成随机的对称密钥，用服务器公钥加密后发送给服务器。
  4. 服务器 用私钥解密，得到相同的对称密钥。
  5. 之后双方使用该对称密钥进行加密通信。

存在的问题

若公匙直接明文传输公钥，可能被中间人伪造。
因此 TLS 通过 数字证书（CA 签名） 验证公钥的真实性，防止中间人攻击。

为什么明文传输公匙会有中间人攻击

假设攻击者在中间拦截了你和服务器之间的通信：

你 ←→ 攻击者 ←→ 服务器

他可以：

  1. 拦截服务器发来的公钥；
  2. 换成自己生成的一对公钥/私钥，发给你；
  3. 你以为是服务器的公钥，实际是攻击者的；
  4. 之后你发的加密数据，攻击者能用自己的私钥解密；
  5. 他再用服务器公钥重新加密转发——你和服务器都没发现被劫持。

这就是典型的中间人攻击

CA证书

CA 证书是什么

CA (Certificate Authority) 是数字证书，用来证明网站的身份真实性，防止公钥在传输中被篡改或伪造。

工作原理

证书签发过程

  1. 网站生成一对密钥（公钥 + 私钥）。
  2. 将公钥 + 域名等信息打包成 CSR 申请文件 ，提交给 CA。
  3. CA 验证网站身份（确认确实是 bank.com）。
  4. CA 使用 CA 私钥 对证书内容的 hash 摘要 进行加密，生成 数字签名。
    ✅ 生成的证书包括：网站公钥、域名信息、CA 签名。

注意： 签名只是对 hash 摘要加密，而不是加密整个证书。

浏览器的验证过程

  1. 服务器把证书发给浏览器。
  2. 浏览器提取证书中的：
    • 公钥信息
    • CA 的数字签名
  3. 浏览器使用 CA 公钥 (系统内置) 解密签名，得到摘要 A。
  4. 再对证书明文部分重新做 hash，得到摘要 B。
  5. 比较 A 和 B：
    • 相同→ 证书未被修改，是合法的
    • 不同→ 证书被篡改，拒绝连接。

什么是工厂模式

工厂模式就是将创建对象的过程封装起来，让使用者只关心对象的使用而不关心它是如何被创建的

简单示例

🌰 场景：我要做一个“发消息”的程序

有两种消息类型：Email 和 SMS。

1️⃣ 定义消息接口（共同规范）

2️⃣ 两种消息的实现类

3️⃣ 工厂类（封装创建逻辑）

4️⃣ 使用工厂创建对象

👉** 总结一句话：**
你不需要 new EmailMessage()，工厂帮你决定创建哪一个，这样以后要扩展新类型（比如微信消息）就只需改工厂，不影响使用方。

工厂模式分类

模式类型 核心思想 结构特点 优点 缺点 适用场景
简单工厂（Simple Factory） 由一个工厂类集中创建所有产品 一个工厂类 + 多个产品类 简单易用，统一创建入口 新增产品需修改工厂代码，不符合开闭原则 产品种类少、变化不频繁
工厂方法（Factory Method） 每个产品由各自的工厂类创建 抽象工厂接口 + 多个具体工厂 + 多个产品类 新增产品无需修改原代码，扩展性好 类数量多、结构更复杂 产品类型多、创建逻辑复杂
抽象工厂（Abstract Factory） 生产一组相关或依赖的产品族 抽象工厂接口 + 多个具体工厂 + 多个产品族 可一次创建多个配套产品，保持一致性 难以扩展新产品族（需改接口） 需要创建多个“配套产品”的系统（如主题样式、品牌设备）

简单工厂

核心思想：
用一个「工厂类」统一负责创建所有对象。

客户端 → 简单工厂 → 产品对象


示例

✅ 优点： 创建逻辑集中、使用简单
⚠️ 缺点： 新增产品要改工厂类 → 不符合开闭原则

适合：小项目、产品种类不多的场景

工厂方法

核心思想：
每种产品一个对应的工厂，让子类决定创建哪种对象。
也就是说——“创建”这件事不再由一个大工厂决定，而是交给不同的小工厂。

客户端 → 抽象工厂接口 → 各具体工厂 → 具体产品


示例

✅ 优点： 新增产品只需新增工厂类，不改旧代码（符合开闭原则）
⚠️ 缺点： 类变多了（一个产品对应一个工厂）

适合：产品类型多、变化频繁的中大型项目

抽象工厂

核心思想：
生产一族相关的产品（不仅仅是一种）。
比如同时生产 “华为手机 + 华为耳机”，或 “苹果手机 + 苹果耳机”。

客户端 → 抽象工厂接口 → 具体工厂A / 具体工厂B → 多个产品族


✅ 优点： 一次创建多个“配套产品”，保持风格一致
⚠️ 缺点： 扩展新产品族（比如“手表”）需要修改所有工厂接口

适合：需要生产多个关联对象的复杂系统（如 UI 主题、品牌设备）

工厂方法和抽象工厂的主要区别是什么？

工厂方法是“一个工厂生产一种产品”；抽象工厂是“一个工厂生产一整套相关产品”。

什么是一致性hash

假设有一个KV（key-value）缓存系统，某个key应该到哪个或者哪些节点上获得，应该是确定的。

因此我们可以使用hash算法，如果客户端要获取指定 key 的数据，通过下面的公式可以定位节点：

hash(key) % 3


但是有一个很致命的问题，如果节点数量发生了变化，也就是在对系统做扩容或者缩容时，必须迁移改变了映射关系的数据，否则会出现查询不到数据的问题。

一致性哈希算法就很好地解决了分布式系统在扩容或者缩容时，发生过多的数据迁移的问题。

一致哈希算法也用了取模运算，但与哈希算法不同的是，哈希算法是对节点的数量进行取模运算，而一致哈希算法是对 2^32（不一定，环的大小可以任意） 进行取模运算，是一个固定的值。

一致性哈希要进行两步哈希：

  • 第一步：对存储节点进行哈希计算，也就是对存储节点做哈希映射，比如根据节点的 IP 地址进行哈希；
  • 第二步：当对数据进行存储或访问时，对数据进行哈希映射；

接着，对要查询的 key-01 进行哈希计算，确定此 key-01 映射在哈希环的位置，然后从这个位置往顺时针的方向找到第一个节点，就是存储该 key-01 数据的节点。

假设节点数量从3增加到4，新的节点D经过hash计算后映射到了下图中的位置

你可以看到，key-01、key-03 都不受影响，只有 key-02 需要被迁移节点 D。

假设节点数量从 3 减少到了 2，比如将节点 A 移除：

你可以看到，key-02 和 key-03 不会受到影响，只有 key-01 需要被迁移节点 B。

因此，在一致哈希算法中，如果增加或者移除一个节点，仅影响该节点在哈希环上顺时针相邻的后继节点，其它数据也不会受到影响。

一致性hash 有什么问题

存在节点分布不均匀的问题，如下图

如何解决

一致性 Hash 引入 虚拟节点，为每个真实节点创建多个映射点，使节点在 Hash 环上的分布变得更均匀，从而解决真实节点少时分布不均的问题，并提高系统在节点增删时的稳定性。

为什么需要虚拟节点？

真实节点太少时，映射到哈希环上的位置不均匀，导致：

  • 请求落点集中在少数节点 → 负载不平衡
  • 节点上下线时，影响区间太大 → 系统不稳定

真实节点少时的“一次性分布不均”问题。

一.CAP理论

CAP是三个英文单词得到缩写

字母 全称 中文含义 说明
C Consistency 一致性 所有节点在同一时间看到的数据是一致的
A Availability 可用性 每个请求都能在有限时间内得到响应（不一定是最新数据）
P Partition tolerance 分区容错性 系统能在网络分区（节点之间通信中断）时继续提供服务

CAP理论的核心结论就是，在分布式系统中，最多只能同时满足C,A,P三者中的两个

例如

假设你有两个节点的分布式数据库：

  • 场景：网络断开（分区出现）
  • 用户 A 向节点1写入数据 X=100；
  • 用户 B 向节点2读取数据。

这时必须选择:

  1. 一致性（C）：节点2暂时不能读到数据（拒绝请求），系统不可用（A失效）
  2. 可用性（A）：节点2马上返回旧数据X=50，系统不一致（C失效）

二.三种典型的取舍模型

类型 取舍 特点 示例
CP 系统 保证一致性和分区容错，放弃可用性 网络异常时牺牲可用性（部分请求失败） HBase、ZooKeeper
AP 系统 保证可用性和分区容错，放弃强一致性 网络异常时允许返回旧数据（最终一致） Cassandra、Eureka
CA 系统 保证一致性和可用性，放弃分区容错 理论上只存在于单机系统 单节点数据库（如本地 MySQL）

在实际系统中更倾向于选择AP+最终一致性

什么是DMA

DMA（Direct Memory Access）允许 I/O 设备在 CPU 不参与数据搬运的情况下，直接访问内存，实现“设备 ↔ 内存”的高速传输。
CPU 只负责发起和结束，不负责搬运。

什么是零拷贝

零拷贝技术让数据在内核态内部直接流动，由 DMA 完成搬运，避免用户态与内核态之间的数据复制，从而减少 CPU 开销并提升 I/O 性能。

什么是虚拟内存

虚拟内存 = 让每个进程看到一片连续的大地址空间，并由操作系统自动把它映射到真实、碎片化的物理内存中，从而解决内存不够用、内存碎片和进程隔离的问题。

虚拟内存主要有两种形式一种是内存分段（早期提出，现在很少使用），一种是内存分页（目前普遍流行的）

内存分段

虚拟内存 = 让每个进程看到一片连续的大地址空间，并由系统自动把它映射到真实、碎片化的物理内存中，从而解决内存不够用、内存碎片和进程隔离的问题。

分段划分，本质上就是通过一个段表进行虚拟地址->真实地址的映射

分段划分会有两个问题

  • 内存碎片
  • 内存交换的效率低

内存碎片

我们来看看这样一个例子。假设有 1G 的物理内存，用户执行了多个程序，其中：

  • 游戏占用了 512MB 内存
  • 浏览器占用了 128MB 内存
  • 音乐占用了 256 MB 内存。

这个时候，如果我们关闭了浏览器，则空闲内存还有 1024 - 512 - 256 = 256MB。

如果这个 256MB 不是连续的，被分成了两段 128 MB 内存，这就会导致没有空间再打开一个 200MB 的程序。

分段会产生“外部碎片”，不会产生“内部碎片”。

  • 不会产生内部碎片：
    分段按“真实需求大小”整体分配，只有找到能容纳整个段的连续空间才会分配，因此不会出现“分配多余空间被浪费”的情况。
  • 会产生外部碎片：
    由于段大小不固定、必须连续，内存中会形成许多小的、彼此不连续的空洞，可能导致后续大的段无法放入，进而产生外部碎片。

对于内存分段的外部内存碎片问题，可以使用内存交换来解决

可以把音乐程序占用的那 256MB 内存写到硬盘上，然后再从硬盘上读回来到内存里。不过再读回的时候，我们不能装载回原来的位置，而是紧紧跟着那已经被占用了的 512MB 内存后面。这样就能空缺出连续的 256MB 空间，于是新的 200MB 程序就可以装载进来。

内存交换效率低

由于内存分段很容易产生外部内存碎片，所以我们不得不进行大量的内存交换，而内存交换涉及到磁盘io效率很低

内存分页

分页是把整个虚拟和物理内存空间切成一段段固定尺寸的大小。这样一个连续并且尺寸固定的内存空间，我们叫页（Page）。在 Linux 下，每一页的大小为 <font style="color:rgb(71, 101, 130);background-color:rgba(27, 31, 35, 0.05);">4KB</font>。虚拟地址与物理地址之间通过页表来映射，如下图：

页表是存储在内存里的，内存管理单元 （MMU（计算机硬件））就做将虚拟内存地址转换成物理地址的工作。

分页是怎么解决分段的「外部内存碎片和内存交换效率低」的问题？

1. 分页不会有外部碎片
内存被预先划分成大小固定的页（如 4KB），页与页紧密排列，因此不会出现“不连续的小空洞”。

2. 分页会有内部碎片
内存分配的最小单位是页，即使程序只用到几百字节，也必须分配整页，所以会产生内部碎片。

3. 分页的换入换出效率更高
当内存不足时，操作系统只需将“近期未使用的少量页面”换出到磁盘，再按需将这些页面换入，而不是移动整个大段，因此交换开销更小。

4. 分页支持按需加载
程序启动时无需一次性加载所有内容，只需建立虚拟页与物理页的映射，程序运行到需要某个虚拟页时，再触发缺页中断并把对应页加载到内存即可。

分页机制下，虚拟地址和物理地址是如何映射的？

在分页机制下，虚拟地址分为两部分，页号和页内偏移。页号作为页表的索引，页表包含物理页每页所在物理内存的基地址，这个基地址与页内偏移的组合就形成了物理内存地址，见下图。

![](https://cdn.nlark.com/yuque/0/2025/png/50543647/1763612066346-bc916948-fa43-48be-8fe2-b1a0ff5cb937.png)

总结一下，对于一个内存地址转换，其实就是这样三个步骤：

  • 把虚拟内存地址，切分成页号和偏移量；
  • 根据页号，从页表里面，查询对应的物理页号；
  • 直接拿物理页号，加上前面的偏移量，就得到了物理内存地址。

多级页表

对于只有一个页表的简单分页

  • 每个进程都有独立的虚拟地址空间，因此每个进程都需要自己的页表。
  • 以 32 位系统为例：虚拟空间 4GB、页大小 4KB → 需要约 100 万个页。
  • 每个页表项 4 字节 → 一个完整页表大小约 4MB。
  • 一个进程 4MB 看似不大，但 100 个进程就要 400MB 内存仅用于存页表。
  • 在 64 位环境下虚拟地址空间更大，页表会更巨大。

为了解决空间问题，就引入了多级页表

对于32位操作系统我们把这个 100 多万个「页表项」的单级页表再分页，将页表（一级页表）分为 <font style="color:rgb(71, 101, 130);background-color:rgba(27, 31, 35, 0.05);">1024</font> 个页表（二级页表），每个表（二级页表）中包含 <font style="color:rgb(71, 101, 130);background-color:rgba(27, 31, 35, 0.05);">1024</font> 个「页表项」，形成二级分页。如下图所示：

分了二级表，映射 4GB 地址空间就需要 4KB（一级页表）+ 4MB（二级页表）的内存，这样占用空间不是更大了吗？

  • 虽然虚拟地址空间是 4GB，但进程实际只使用其中很小的一部分，所以绝大多数页表项始终是空的。
  • 如果只使用一级页表，就必须一次性构建覆盖整个 4GB 空间的页表（约 4MB），而且每个进程都要有一份，这会造成巨大浪费。
  • 二级页表的做法是：一级页表仍覆盖整个虚拟空间，但只有当某个一级页表项被访问时，才按需创建对应的二级页表。
  • 一级页表始终存在（大小约 4KB）
  • 二级页表“用到哪一块虚拟空间就创建哪一块”，未访问的区域不占用页表内存

对于64位操作系统我们一般会使用四级页表

为什么不分级的简单页表不能按需创建？

因为简单页表承担的是“直接把虚拟页号映射到页表项”的工作，它必须完整存在，否则 MMU 无法完成地址转换。 所以页表一定要覆盖全部虚拟地址空间

TLB

根据局部性原理，程序在短时间内会频繁访问相同的地址区域。所以在cpu 芯片中加入了一个专门存放程序最常访问的页表项的 Cache，这个 Cache 就是 TLB（Translation Lookaside Buffer） ， 通常称为页表缓存、转址旁路缓存、快表等。

CPU 内部集成了内存管理单元（MMU），负责地址转换，并与 TLB 协同工作。

有了 TLB 后，CPU 查地址时会先查 TLB；如果命中，则无需访问页表；如果未命中，再去查正常的页表。

由于程序具有局部性，常访问的页通常只有少数几个，所以 TLB 的命中率往往非常高。
