什么是代理对象

代理对象就是一个“替身对象”，它包裹了你的原始对象（目标对象），拦截你对目标对象的调用，在前后插入一些额外逻辑。

在 Java 中，这种代理对象通常由两种机制实现：

  1. JDK 动态代理（JDK Proxy）
  2. CGLIB 字节码代理

Spring 会根据 Bean 的情况自动选择其中一种。

为什么 Spring 要创建代理对象？

为了实现AOP（面向切面编程）、事务控制、安全检查、缓存注解等等。这些特性都需要在方法执行前后“插入逻辑”，而不修改原代码。

例如

在 Spring 中，@Transactional 依靠 AOP 代理机制 实现。

当我们调用：userService.saveUser();时，实际上并不是调用的原始的UserService，而是Spring 创建的代理对象，这个代理对象会在方法执行的前后自动插入事务逻辑

动态代理和静态代理的区别

  1. 静态代理是指：代理类在编译时就已经存在，由程序员手写或编译生成。也就是说，你要先写一个接口、一个真实类（被代理类），再写一个代理类，在里面调用真实类的方法。
  2. 动态代理是指：代理类在运行时动态生成，而不是编译时写死的。

什么是Spring 三级缓存

Spring 的三级缓存是用来提前暴露未完全创建的 Bean 对象，从而解决单例 Bean 的循环依赖问题的机制。

例如

A 依赖 B，B 又依赖 A。
👉 如果直接反射 new，就会死循环。

所以 Spring 必须想办法“先创建一半”，把引用提前暴露出去。这就是三级缓存登场的地方。

三级缓存结构 （核心类 DefaultSingletonBeanRegistry）

Spring 在这个类里维护了三个Map

缓存层级 名称 作用 何时存放 何时清除
一级缓存 singletonObjects 已经完全初始化好的单例 Bean（可直接使用） Bean 创建完成后放入 容器关闭时清空
二级缓存 earlySingletonObjects 提前曝光的 Bean 实例（已 new 但未完全初始化） 遇到循环依赖时提前暴露 创建完成后清空
三级缓存 singletonFactories 可以创建“早期对象”的工厂（ObjectFactory） 实例化后、属性注入前放入 当对象被取出或转移到二级缓存后清空

创建对象过程

1️⃣ 创建 A

  • 通过反射调用构造器实例化出 A（实例化阶段完成，但还未注入属性）。
  • 注册 A 的早期对象工厂：
  • singletonFactories[“A”] = () -> getEarlyBeanReference(“A”, A实例）
  • （这一步只是把一个“可以返回 A 早期引用的工厂函数”放入三级缓存）
  • 进入注入A属性阶段，发现依赖B->执行getBean(“B”) 去创建B

2️⃣ 创建 B（因为 A 依赖 B）

  • 通过反射调用构造器实例化出 B（实例化阶段完成）。
  • 注册 B 的早期对象工厂：
  • singletonFactories[“B”] = () -> getEarlyBeanReference(“B”, B实例)
  • 进入注入B属性阶段，发现依赖A->执行getBean(“A”)

3️⃣ 在创建 B 的过程中注入 A

调用 getBean("A") 时，Spring 依次查找：

  • 一级缓存（singletonObjects）：❌ 没有
  • 二级缓存（earlySingletonObjects）：❌ 没有
  • 三级缓存（singletonFactories）：✅ 有 “A 的工厂”
  • 调用该工厂： 生成 A 的早期引用（可能是原始 A，也可能是 AOP 代理对象）。 将这个早期 A 放入二级缓存：
  • 使用这个早期 A 给 B 注入依赖。

4️⃣ B 创建完成

  • B 的依赖注入与初始化都完成 → 放入一级缓存：
  • 清理 B 的二级、三级缓存记录。

5️⃣ 回到 A，完成剩余步骤

  • 继续给 A 注入 B（此时 B 已在一级缓存中，可以直接注入）。
  • A 完成依赖注入与初始化 → 放入一级缓存：
  • 清理 A 的二级、三级缓存记录。

  1. A 实例化 → 放入二级缓存

  3. 然后 A 完成初始化（此时 AOP 代理可能介入）→ A 最终变成代理对象。

一.简要概括

好的，下面是对 Spring Bean 生命周期四个阶段的一句话概括：

  1. 实例化（Instantiation）：Spring 通过反射创建 Bean 对象的实例。
  2. 属性赋值（Populate Properties）：Spring 将配置文件或注解中定义的依赖注入到 Bean 的属性中。（三级缓存）
  3. 初始化（Initialization）：Spring 调用 Bean 的初始化方法（如 @PostConstruct、afterPropertiesSet() 或自定义 init-method）进行初始化逻辑。
  4. 销毁（Destruction）：当容器关闭时，Spring 调用销毁方法（如 @PreDestroy、destroy() 或自定义 destroy-method）释放资源。

二.初始化

初始化阶段的目标：让 Bean 完全准备好可以使用。 经过第一二步 反射出来的对象现在还只是“一个有属性的普通对象”；Spring 要在初始化阶段执行一系列“钩子（回调）”，让 Bean 真正“活起来”。

初始化阶段执行顺序

1️⃣ 实例化（new 对象）

2️⃣ 属性注入（依赖注入）

3️⃣ 调用 Aware 接口（如 BeanNameAware）

4️⃣ BeanPostProcessor 前置处理

➡️ 5️⃣ @PostConstruct（执行初始化逻辑）

6️⃣ afterPropertiesSet() —— 来自 InitializingBean 接口

7️⃣ init-method() —— 在 XML / @Bean 中指定

8️⃣ BeanPostProcessor 后置处理（常用于 AOP 代理）


Bean的初始化方法

@PostConstruct（重点 ⭐）

  • 作用：在 Bean 完成依赖注入后、但在使用前自动执行一次初始化逻辑。
  • 执行时机：依赖注入完成 → 初始化前（在 AOP 代理之前）。

@PostConstructor 方法要求

  • 返回值类型：void
  • 无参数（不能带任何参数）
  • 不能抛出受检异常（checked exception）

afterPropertiesSet 与 init-method（了解）

方法 定义方式 调用时机 场景
afterPropertiesSet() 实现 InitializingBean
接口		 @PostConstruct 之后 用于框架级初始化
init-method() 在 XML 或 @Bean 指定 afterPropertiesSet 之后 用于配置文件初始化

三者都能定义初始化逻辑，执行顺序为：
@PostConstruct → afterPropertiesSet → init-method

aware 相关接口就是在告诉bean自己是谁（BeanNameAware），自己在哪（ApplicationContextAware），谁创建了自己（BeanFactoryAware）

** AOP（面向切面编程）是一种将公共逻辑抽取并封装为切面的编程思想，它能够在需要的地方自动切入，从而实现代码复用，同时保持业务代码的简洁与清晰。**

  1. 优点：
  • 代码解耦：公共逻辑与业务逻辑分离，业务代码更干净。
  • 复用性强：一个切面可以应用到多个方法，减少重复代码。
  • 灵活扩展：新增公共逻辑时，只需修改切面，无需改动业务代码。
  1. 缺点：
  • 调试复杂：切面逻辑可能影响多个方法，错误时需要追踪切面代码。
  • 性能开销：AOP通过动态代理实现，大量切面可能增加方法调用耗时（但日常业务中可忽略）。

核心概念

  1. 切面(Aspect)： 编写执行逻辑,里面定义拦哪些点（ @Pointcut ）,做哪些事（@Advice）
  2. 切入点：要切入的方法
  3. 通知: 在切入点执行的逻辑

类内调用失效的本质原因是：AOP 依赖代理生效，而 this.xxx() 是直接调用目标对象本身，不经过代理，自然无法触发 AOP。

可以有以下两种解决方案

将方法抽离为bean

打开exposeProxy

在启动类或任何一个配置类上加： @EnableAspectJAutoProxy(exposeProxy = true)

在类内调用this.xxx方法时先获得这个类的proxy 代理类

内部实现

AopContext.currentProxy() 本质上就是从当前线程的 ThreadLocal 中读取“当前正在执行的代理对象”。
Spring 会在进入代理方法时将该代理写入 ThreadLocal，方法执行完再恢复，因此不同线程互不影响。

RocketMQ消费重试机制

在 RocketMQ 中，Broker 将消息发送给消费者后不会立即删除消息，而是保留在 CommitLog 中；仅当消费者成功确认（ACK）后，才认为消费完成。若消费者处理消息失败，Broker 会将消息放入 重试队列，并根据延迟级别多次重投；若超过最大重试次数仍未成功，则消息会被转入 死信队列（DLQ），供人工排查与补偿处理。

应用场景

RocketMQ 消费重试机制用于处理偶发性的业务消费失败，以保证消息最终一致性；
不应用于流程分支或限流控制，否则会导致消息堆积与系统性能下降。

正确应用场景

场景 说明
1. 业务逻辑暂时无法完成 比如依赖的下游服务未准备好、关联事务状态未更新、依赖数据尚未写入数据库等。消息重试可在一段时间后自动重新消费，最终成功。
2. 消费失败为偶发性错误 如网络抖动、瞬时资源不可用、数据库锁冲突等临时异常，短暂延迟后再次消费通常能恢复正常。
3. 需要保证业务最终一致性 在消息驱动的异步系统中，通过重试机制补偿异常情况，确保消息最终被成功处理。

错误应用场景

场景 不合理点
1. 用消费失败做业务条件分支 如果消费逻辑中主动让部分消息“失败”以触发不同分支流程，这属于滥用重试；应通过业务字段或消息路由实现逻辑分流。
2. 用消费失败实现限流/削峰 消费失败会触发重投而不是减速，会造成消息积压与不必要的 Broker 压力。正确做法是使用 RateLimiter 或队列削峰机制。
3. 预期失败为常态 如果多数消息都会消费失败（例如逻辑未准备好），反复重试只会增加系统负担，建议通过延迟消息或消息过滤机制来解决。

消费重试的触发条件

消费失败，包括消费者返回消息失败状态标识或抛出非预期异常。

在 RocketMQ 中，消费失败（Consume Failure） 指的是消费者在收到消息后，没有正确处理完业务逻辑， Broker 因此认为这条消息“未被成功消费”。

消息处理超时，包括在PushConsumer中排队超时。

消费超时 是指消费者在 Broker 规定的时间内未返回消费结果（成功或失败）。
由于 Broker 需要确认每条消息的消费状态，如果消费者长时间未响应，Broker 会认为该消息超时未完成消费，并将其判定为消费失败。
需要注意，消息超时并不等同于代码出错，它通常是由于消费者处理速度过慢、线程阻塞或业务逻辑执行过长等原因导致的。

消息超时触发条件

场景 说明
🕒 处理太慢 消费逻辑执行时间超过了 Broker 允许的上限（默认 15 分钟）
🧵 线程排队超时 PushConsumer 模式中，消息在客户端线程池等待太久（线程繁忙）
🧱 卡死或阻塞 消费者业务逻辑中死循环、网络阻塞、锁等待等，导致迟迟不返回结果

消息重试策略主要行为

消息重试策略主要决定了当消息消费失败后，RocketMQ 如何重新投递消息。
它包含以下几个关键行为：

  • 重试状态流转：描述消息在重试过程中的状态变化，比如“消费失败 → 等待重试 → 再次投递”。
  • 重试间隔时间：指每次消费失败后，Broker 等待多长时间再重新发送这条消息。
  • 最大重试次数：规定同一条消息最多可以被重新消费的次数，超过上限就会被放入死信队列（DLQ）。

消息重试策略差异

根据消费者类型不同，消息重试策略的具体内部机制和设置方法有所不同，具体差异如下

消费者类型 重试过程状态机 重试间隔 最大重试次数
PushConsumer 已就绪 → 处理中 → 待重试 → 提交（成功） / 死信（失败） 无序消息：采用阶梯式延迟重试（10s、30s、1m、2m…）顺序消息：使用固定间隔时间重试 由消费者分组元数据控制（默认 16 次，可通过参数配置）
SimpleConsumer 已就绪 → 处理中 → 提交（成功） / 死信（失败） 通过 API 调整消息的“不可见时间（Invisible Duration）”来自定义重试间隔 同样由消费者分组元数据控制（默认 16 次）

说明补充：

  • PushConsumer：框架自动管理重试流程，开发者只需返回状态或抛异常即可触发重试。
  • SimpleConsumer：开发者完全掌控重试逻辑，可灵活调整“消息重新可见”的时间，实现自定义延迟重投。

PushConsumer 消费重试策略

PushConsumer消费消息时，消息的几个主要状态如下：

  • Ready：已就绪状态。消息在Apache RocketMQ服务端已就绪，可以被消费者消费。
  • Inflight：处理中状态。消息被消费者客户端获取，处于消费中还未返回消费结果的状态。
  • Commit：提交状态。消费成功的状态，消费者返回成功响应即可结束消息的状态机。
  • WaitingRetry 是 PushConsumer 独有的状态。当消费者处理消息失败或发生消费超时时，RocketMQ 会触发重试逻辑：如果当前消息的重试次数还没达到上限，消息就会进入 待重试状态。在等待一段时间（即重试间隔）后，消息会重新变为 已就绪状态，再次投递给消费者。通过设置合适的重试间隔，可以避免消息频繁失败导致系统高频重试，从而提升系统稳定性。
  • DLQ：死信状态。消费逻辑的最终兜底机制，若消息一直处理失败并不断进行重试，直到超过最大重试次数还未成功，此时消息不会再重试，会被投递至死信队列。您可以通过消费死信队列的消息进行业务恢复。

最大重试次数

PushConsumer的最大重试次数由消费者分组创建时的元数据控制，例如，最大重试次数为3次，则该消息最多可被投递4次，1次为原始消息，3次为重试投递次数。
