介绍一下java的缓存池机制

这个机制的核心思想是：对于一些频繁被使用、不可变的对象，将它们缓存起来重复利用，从而避免频繁地创建和销毁对象，以提高性能和节省内存。

最主要的两种内置缓存池是 字符串常量池 和 包装类缓存池

  1. 字符串常量缓存池

这是 Java 中最著名、也最特殊的一个缓存机制。

  • 作用：存储字符串字面量（String literals）和通过 String.intern() 方法添加的字符串。
  • 位置：在 JDK 7 之后，字符串常量池从永久代（PermGen）移到了**堆内存（Heap）**中。
  • 工作机制：
    1. 当你通过字面量的方式创建一个字符串时（例如 String s1 = "Java";），JVM 会先去字符串常量池里检查是否存在内容为 “Java” 的字符串。
    2. 如果存在，JVM 不会创建新对象，而是直接返回池中该对象的引用给 s1。
    3. 如果不存在，JVM 会在池中创建一个新的 “Java” 对象，然后将其引用返回给 s1。
    4. 当你使用 new String("Java") 时，new 关键字会强制在堆内存中创建一个全新的对象，这个对象不在常量池中。因此建议构建字符串常量的都是不要通过new来构建,而是通过字面量来创建字符串,因为这样可以有被缓存的概率，性能更高

2. 包装类缓存池 (Wrapper Class Caching)

Java 为一些基本数据类型的包装类也提供了缓存池。这个机制主要在**自动装箱（Autoboxing）**或手动调用 valueOf() 方法时生效。

a) 整型包装类 (Integer, Long, Short, Byte)

这些类会缓存一个特定范围内的对象。

  • 缓存范围：
    • Byte, Short, Long：固定缓存 -128 到 127 之间的值。
    • Integer：默认缓存 -128 到 127 之间的值。这个范围的上界是可以通过 JVM 参数 -XX:AutoBoxCacheMax=<size> 来调整的。

    • 自动装箱：Integer i = 100;
    • 调用 valueOf() 方法：Integer i = Integer.valueOf(100);

线程池ThreadPoolExecutor

分类于 Java

线程池的基本工作流程

corePoolSize->workQueue->maximumPoolSize

1.线程池先看当前运行的线程数是否小于 corePoolSize

  • 如果是 → 直接创建新线程执行任务（即“核心线程”）。
  • 如果否 → 进入下一步。
  1. 如果线程数 ≥ corePoolSize，就尝试把任务放进队列 workQueue
  • 如果队列没满 → 任务进入队列等待。
  • 如果队列满了 → 进入下一步。
  1. 如果队列已满，并且线程数 < maximumPoolSize
  • 再创建新线程（非核心线程）来执行任务。
  1. 如果线程数已经达到 maximumPoolSize，且队列也满
  • 就会触发 拒绝策略 handler，决定怎么处理任务（丢弃、抛异常、调用者运行等）。
  1. 线程空闲时，如果超过 keepAliveTime
  • 非核心线程会被销毁；
  • 若设置了 allowCoreThreadTimeout = true，核心线程也可能销毁。

线程池的核心参数

  1. corePoolSize（必需）：核心线程数。默认情况下，核心线程会一直存活，但是当将 allowCoreThreadTimeout 设置为 true 时，核心线程也会超时回收。

    注意一开始并不会创建核心线程，只有当执行execute的时候的时候才会创建核心线程,如果 想让核心线程在创建线程池时就全部启动 （常用于高并发预热场景）需要调用prestartAllCoreThreads()方法

  • maximumPoolSize（必需）：线程池所能容纳的最大线程数。当活跃线程数达到该数值后，后续的新任务将会阻塞。maximumPoolSize包括corePoolSize
  • keepAliveTime（必需）：线程闲置超时时长。如果超过该时长，非核心线程就会被回收。如果将 allowCoreThreadTimeout 设置为 true 时，核心线程也会超时回收。
  • unit（必需）：指定 keepAliveTime 参数的时间单位。常用的有：TimeUnit.MILLISECONDS（毫秒）、TimeUnit.SECONDS（秒）、TimeUnit.MINUTES（分）。
  • workQueue（必需）：任务队列。通过线程池的 execute() 方法提交的 Runnable 对象将存储在该参数中。其采用阻塞队列实现。
  • threadFactory（可选）：线程工厂。用于指定为线程池创建新线程的方式。
  • handler（可选）：拒绝策略。当达到最大线程数时需要执行的拒绝策略。

什么是阻塞队列什么是非阻塞队列

  1. 阻塞队列:阻塞队列是一种支持线程等待的队列 当队列满了或者空了时，它会让线程“暂时休眠”，直到条件满足再继续执行。
  2. 非阻塞队列： 非阻塞队列不会让线程等待，它只会立即返回结果。

比如：

  • 如果队列满了，offer() 返回 false；
  • 如果队列空了，poll() 返回 null。

它不会让线程睡眠、等待条件变化。

线程池通过execute或submit 方法提交任务

execute（）

  • 只能提交 Runnable
  • 没有返回值
  • 如果任务抛异常 → 异常会直接在控制台打印（线程池中的线程抛出）

Future 和FutureTask的关系

Future是接口只代表一个结果，可以查询结果，取消任务，本身不能运行任务

FutureTask 是类实现了Runnable和Future 接口，因此它可以Future->get()，cancel 查询结果取消任务,

Runnable->可以运行任务，所以FutureTask是一个可以被执行的任务容器

四大引用

分类于 Java

JAVA中有四种引用类型，分别是强引用，软应用，弱引用，虚引用

一、强引用（Strong Reference）

  • 定义：最常见的引用，使用 new 创建。
  • 特性：只要有强引用存在，对象就不会被回收；即使内存不足，也只会抛出 OutOfMemoryError。

二、软引用（Soft Reference）

  • 定义：当内存不足时，GC 会回收软引用对象。
  • 用途：常用于实现缓存。

三、弱引用（Weak Reference）

  • 定义：只要发生 GC，就会回收对象（无论内存是否足够）。
  • 用途：常用于ThreadLocal、WeakHashMap 等。

四.虚引用

引用队列（ReferenceQueue）

在理解虚引用前，先要了解 引用队列（ReferenceQueue）。
当某个引用（软、弱、虚）所指向的对象被判定可回收时，JVM 会将“引用对象本身”（而非目标对象）加入该队列。
不同的是，软引用和弱引用在对象被标记为可回收时入队，而虚引用则在对象真正被回收后才入队。

什么时虚引用

虚引用（PhantomReference） 是 Java 提供的一种最弱的引用类型。
它的最大特点是：

  • 不能通过 **get()** 访问对象（get访问返回值总为null）
  • 只用来感知对象被真正回收的时机

为什么需要虚引用

我们知道：

  • 软引用：用于缓存，内存不足时清理；
  • 弱引用：用于弱依赖对象，GC 一来就清理；
  • 但这两种引用入队太早（对象还没真正销毁）。

有些资源（比如：文件句柄、堆外内存、Socket 连接）只有在对象彻底销毁时才安全释放。
这时候就要用虚引用。

CountDownLatch

分类于 JUC ， 锁

CountDownLatch 是 Java 中用于实现线程同步的一种工具，它的作用是允许一个或多个线程等待其他线程完成某些操作后再继续执行。

一.基本概念

CountDownLatch 的名字可以直译为“倒计时锁”，它通过维护一个计数器来实现线程之间的等待。这个计数器会被其他线程逐步减少，直到变为零，表示所有等待的操作完成，调用 await() 方法的线程才会被释放，可以继续执行。

二.工作原理

  • CountDownLatch 在创建时会设置一个初始的计数值，这个计数值通常表示需要等待的事件或操作的数量。
  • 每当一个线程完成了它的任务，它就会调用 countDown() 方法将计数器减1。
  • 当计数器变为零时，所有调用 await() 方法的线程会被唤醒，继续执行。

三.常用方法

  1. 构造方法
    • CountDownLatch(int count): 创建一个 CountDownLatch 对象，初始化计数器为 count。
  2. **await()**** 方法**:
    • await(): 当前线程会等待，直到计数器为 0。线程会被阻塞，直到计数器减为零。
    • await(long timeout, TimeUnit unit): 当前线程会等待指定的时间，超时后会自动继续执行。
  3. **countDown()**** 方法**:
    • countDown(): 将计数器减1，表示一个线程完成了任务。
  4. **getCount()**** 方法**:
    • getCount(): 返回当前计数器的值。

锁

分类于 JUC
  1. 内置锁（synchronized）

Java 中的 synchronized 关键字是内置锁机制的基础，可以用于方法或代码块。当一个线程进入 synchronized 代码块或方法时，它会获取关联对象的锁；当线程离开该代码块或方法时，锁会被释放。如果其他线程尝试获取同一个对象的锁，它们将被阻塞，直到锁被释放。
其中，synchronized 加锁有无锁、偏向锁、轻量级锁和重量级锁四个级别。

  • 偏向锁：当一个线程进入同步块时，如果没有任何其他线程竞争，就会使用偏向锁，以减少锁的开销。
  • 轻量级锁：使用线程栈上的数据结构，避免了操作系统级别的锁。
  • 重量级锁：涉及操作系统级的互斥锁。

  1. ReentrantLock 是一个显式的锁类，提供了比 synchronized 更高级的功能，如可中断的锁等待、定时锁等待、公平锁选项等。 ReentrantLock 使用 lock() 和 unlock() 方法来获取和释放锁。 公平锁：按照线程请求锁的顺序来分配，保证了锁分配的公平性，但可能增加锁的等待时间。 非公平锁：不保证锁的分配顺序，可以减少锁的竞争，提高性能，但可能造成某些线程的饥饿。

  2. 读写锁（ReadWriteLock） 允许多个读取者同时访问共享资源，但只允许一个写入者。

  3. 自旋锁 自旋锁是一种锁机制，线程在等待锁时会持续循环检查锁是否可用， 而不是放弃 CPU 并阻塞。通常可以使用 CAS（Compare-And-Swap） 来实现。
    在锁等待时间很短的情况下可以提高性能，但过度自旋会浪费 CPU 资源。

CAS

是一种硬件级别的原子操作指令，用来在多线程环境下实现无锁（Lock-Free）并发。

它的思想很简单：
“当且仅当变量的值与我预期的一样时，才把它改成新值；否则不改。”

CAS的执行策略

当 CAS 更新失败时，线程会自旋重试，但长期竞争会导致 CPU 占用过高。 需要注意，CAS 本身没有自旋次数限制或退避机制。 在 Java 中，synchronized（JVM 层）和 ReentrantLock（JUC 层） 都会在竞争较轻的场景下使用 CAS + 自旋机制 实现“轻量级锁”或“自旋锁”优化，并配合 自旋次数限制 与 退避策略 来防止无限自旋造成性能浪费。

  • 自旋次数限制 有的场景中，自旋次数是有限的。 如果自旋超过阈值仍然失败，就会进入阻塞等待（park）。
  • 退避策略 如果 CAS 一直失败，会随机休眠一小段时间再重试， 防止多个线程在同一时间同时抢占 CPU（减少竞争冲突）。

用版本号机制解决CAS的ABA问题

什么是ABA问题

在使用 CAS（Compare-And-Swap）时，只比较“值是否一样”，而不关心这个值是否被改过再改回。
举个例子：

时间 操作线程 变量值
t1 线程1 读取 A A
t2 线程2 把 A 改为 B B
t3 线程2 又把 B 改回 A A
t4 线程1 执行 CAS 认为 A 没变 ✅ 成功

实际上数据被改过，但 CAS 看不出来，因为值又变回了 “A”。
这就可能造成逻辑错误 —— 看似没变，实则被修改过。

2.AtomicStampedReference 的原理

AtomicStampedReference 在 CAS 的基础上引入一个版本号（stamp）。

  • 每次更新时，除了比较值 value，还要比较版本号 stamp；
  • 成功更新后，版本号自动 +1。

因此即使值又变回来了，版本号也会不同，从而避免 ABA 问题。

执行过程

  1. 当前值是 100，版本号是 1
  2. 如果在此期间没有其他线程修改，则 compareAndSet 成功，将：
    • 值改为 101
    • 版本号改为 2
  3. 如果其他线程在此期间修改了值或版本号，则更新失败。

Sychronized 的锁升级

具体的锁升级的过程是：无锁->偏向锁->轻量级锁->重量级锁。

注意：锁只能升级，不会降级。

锁状态 特点 适用场景
无锁（Unlocked） 没有任何线程竞争 单线程执行
偏向锁（Biased Lock） 只被同一个线程使用 几乎无竞争的场景
轻量级锁（Lightweight Lock） 存在短暂竞争（线程交替执行） 低并发
重量级锁（Heavyweight Lock） 多线程并发竞争激烈 高并发
  1. 偏向锁

第一次线程获得锁时：

  • JVM 会在 对象头（Mark Word） 中记录当前线程的 ID；
  • 下次同一线程进入同步块时，只需判断线程 ID 是否匹配即可；
  • 不需要 CAS 操作，也不需要进入 Monitor；
  • 因此几乎 零成本加锁。
  1. 轻量级锁，当一个线程也尝试获取锁时
  • 会在当前线程的栈帧中创建一个锁记录（Lock Record）；
  • 通过 CAS 竞争锁；
  • 如果竞争成功，进入同步块；
  • 如果竞争失败（有人拿着锁），进入自旋（不断尝试获取）。
  1. 重量级锁，当有两个以上的线程获取锁的时候轻量级锁就会升级为重量级锁，因为CAS如果没有成功的话始终都在自旋，进行while循环操作，这是非常消耗CPU的，但是在升级为重量级锁之后，线程会被操作系统调度然后挂起，这可以节约CPU资源。

JVM对Synchronized的优化

synchronized 核心优化方案主要包含以下 4 个：

锁膨胀：synchronized 从无锁升级到偏向锁，再到轻量级锁，最后到重量级锁的过程，它叫做锁膨胀也叫做锁升级。

锁消除：指的是在某些情况下，JVM 虚拟机如果检测不到某段代码被共享和竞争的可能性，就会将这段代码所属的同步锁消除掉，从而到底提高程序性能的目的。

锁粗化：将多个连续的加锁、解锁操作连接在一起，扩展成一个范围更大的锁。

自适应自旋锁：指通过自身循环，尝试获取锁的一种方式，优点在于它避免一些线程的挂起和恢复操作，因为挂起线程和恢复线程都需要从用户态转入内核态，这个过程是比较慢的，所以通过自旋的方式可以一定程度上避免线程挂起和恢复所造成的性能开销。

一.ThreadLocal 为什么会发生内存泄露

  • ThreadLocalMap 的 key（指向 ThreadLocal 对象）是弱引用，一旦没有外部强引用，key 会被 GC 回收；
  • 但 value 是强引用，当 key 被回收后，map 中对应的 entry 仍然保留 value 引用 —— 这会导致 value 无法被回收；
  • 如果线程是 长生命周期（如线程池中的线程），这些“孤儿 entry”会一直占住内存，造成泄露直至 OOM。

synchronized锁升级

synchronized 锁的四种状态

注意synchronized 的锁只会升级不会降级

无锁状态（No Lock）：

  • 说明：对象在没有任何线程竞争的情况下处于无锁状态。此时，任何线程都能轻松访问该对象的方法或代码块。
  • 何时发生：当没有线程访问或竞争锁时，锁处于无锁状态。

偏向锁（Biased Locking）：

  • 说明：当对象的锁长时间没有被其他线程竞争时，JVM 会启用偏向锁。偏向锁会将锁附加到第一个获得锁的线程上，减少锁的获取和释放操作，避免不必要的竞争。
  • 何时发生：当对象只有一个线程访问时，JVM 会为该对象启用偏向锁，锁会”偏向”于第一个获取锁的线程。
  • 升级条件：当其他线程竞争该锁时，偏向锁会升级为轻量级锁。

轻量级锁（Lightweight Locking）：

  • 说明：当有多个线程竞争锁时，JVM 会启用轻量级锁。轻量级锁使用 CAS（Compare-And-Swap） 技术，通过在对象头中记录锁的状态来减少锁的开销，避免线程阻塞。
  • 何时发生：当锁发生竞争时，JVM 会通过 CAS 尝试轻量级锁。如果竞争不激烈且 CAS 成功，线程可以直接获得锁。
  • 升级条件：如果 CAS 操作失败或竞争非常激烈，轻量级锁会升级为重量级锁。

重量级锁（Heavyweight Locking）：

  • 说明：当线程竞争激烈且无法通过轻量级锁解决时，JVM 会使用重量级锁。重量级锁通过操作系统的互斥量（mutex）来管理锁，这会导致线程阻塞和上下文切换，性能较差。
  • 何时发生：当多个线程竞争同一把锁，并且轻量级锁无法解决竞争时，JVM 会升级为重量级锁。
  • 升级条件：轻量级锁竞争失败时，JVM 会使用操作系统的互斥量（mutex）进行锁的管理。

偏向锁

偏向锁的工作原理

  • 当一个线程第一次获得对象的锁时，JVM 会将该对象的锁状态设置为 偏向锁，并记录下获得锁的线程 ID。
  • 之后，如果相同的线程再次访问该对象，不需要重新获取锁，JVM 会直接认为这个线程已经持有锁，因此锁的获取和释放都没有任何开销。
  • 锁的释放：当线程退出同步代码块时，锁会被释放，但锁的状态仍然保持为偏向锁，直到有另一个线程竞争锁。

何时使用偏向锁

偏向锁适用于 没有竞争 的情况，即在大部分时间里，只有一个线程访问该对象。当一个线程获取锁后，偏向锁会继续偏向这个线程，避免每次都要进行加锁的性能开销。

无锁->偏向锁的升级过程

当第一个线程访问同步代码时（也就是synchronized 修饰的代码块），它会 尝试获取对象的锁。此时，JVM 会将对象的锁从无锁状态切换为 偏向锁，并将锁标记为偏向于当前线程。锁的状态会保存当前线程的 ID。

轻量级锁

轻量级锁的工作原理

CAS 操作：轻量级锁通过 CAS（Compare-And-Swap） 技术来实现加锁。它尝试通过原子操作直接改变对象头的锁标志位。

竞争时：当有多个线程竞争锁时，JVM 会让线程通过 自旋（不断尝试）来获取锁，直到成功为止。
