synchronized详解

分类于 JUC

synchronized 是 Java 中的一个关键字，用于实现线程同步，确保同一时间内只有一个线程可以访问某段代码，同时synchronized是可重入锁。

Sychronized的使用方法

修饰实例方法（对象锁）

这种方式锁住的是当前实例对象(this)，同一个实例的多个同步方法会互斥，不同实例的同步方法不会相互干扰。

修饰静态方法（类锁）

这种方式锁住的是类对象(Class对象)，所有实例共享同一把锁，即使不同实例调用静态同步方法也会互斥。

同步代码块（指定锁对象）

注意这种方式锁住的是lock 这个Object 对象实例而不是Object 这个类对象

同步静态代码块（类锁）

这种方式锁住的就是ClassName这个类对象了

如何理解锁住类对象和锁住对象实例

一定不要认为锁住类对象之后就锁住了所有实例，因为静态同步方法锁住的是类本身（**Test.class**），而实例同步方法锁住的是某个具体的对象实例，它们锁定的是不同的资源，因此不会互相干扰。

synchronized锁升级

synchronized 锁的四种状态

注意synchronized 的锁只会升级不会降级

无锁状态（No Lock）：

  • 说明：对象在没有任何线程竞争的情况下处于无锁状态。此时，任何线程都能轻松访问该对象的方法或代码块。
  • 何时发生：当没有线程访问或竞争锁时，锁处于无锁状态。

偏向锁（Biased Locking）：

  • 说明：当对象的锁长时间没有被其他线程竞争时，JVM 会启用偏向锁。偏向锁会将锁附加到第一个获得锁的线程上，减少锁的获取和释放操作，避免不必要的竞争。
  • 何时发生：当对象只有一个线程访问时，JVM 会为该对象启用偏向锁，锁会”偏向”于第一个获取锁的线程。
  • 升级条件：当其他线程竞争该锁时，偏向锁会升级为轻量级锁。

轻量级锁（Lightweight Locking）：

  • 说明：当有多个线程竞争锁时，JVM 会启用轻量级锁。轻量级锁使用 CAS（Compare-And-Swap） 技术，通过在对象头中记录锁的状态来减少锁的开销，避免线程阻塞。
  • 何时发生：当锁发生竞争时，JVM 会通过 CAS 尝试轻量级锁。如果竞争不激烈且 CAS 成功，线程可以直接获得锁。
  • 升级条件：如果 CAS 操作失败或竞争非常激烈，轻量级锁会升级为重量级锁。

重量级锁（Heavyweight Locking）：

  • 说明：当线程竞争激烈且无法通过轻量级锁解决时，JVM 会使用重量级锁。重量级锁通过操作系统的互斥量（mutex）来管理锁，这会导致线程阻塞和上下文切换，性能较差。
  • 何时发生：当多个线程竞争同一把锁，并且轻量级锁无法解决竞争时，JVM 会升级为重量级锁。
  • 升级条件：轻量级锁竞争失败时，JVM 会使用操作系统的互斥量（mutex）进行锁的管理。

偏向锁

偏向锁的工作原理

  • 当一个线程第一次获得对象的锁时，JVM 会将该对象的锁状态设置为 偏向锁，并记录下获得锁的线程 ID。
  • 之后，如果相同的线程再次访问该对象，不需要重新获取锁，JVM 会直接认为这个线程已经持有锁，因此锁的获取和释放都没有任何开销。
  • 锁的释放：当线程退出同步代码块时，锁会被释放，但锁的状态仍然保持为偏向锁，直到有另一个线程竞争锁。

何时使用偏向锁

偏向锁适用于 没有竞争 的情况，即在大部分时间里，只有一个线程访问该对象。当一个线程获取锁后，偏向锁会继续偏向这个线程，避免每次都要进行加锁的性能开销。

无锁->偏向锁的升级过程

当第一个线程访问同步代码时（也就是synchronized 修饰的代码块），它会 尝试获取对象的锁。此时，JVM 会将对象的锁从无锁状态切换为 偏向锁，并将锁标记为偏向于当前线程。锁的状态会保存当前线程的 ID。

轻量级锁

轻量级锁的工作原理

CAS 操作：轻量级锁通过 CAS（Compare-And-Swap） 技术来实现加锁。它尝试通过原子操作直接改变对象头的锁标志位。

竞争时：当有多个线程竞争锁时，JVM 会让线程通过 自旋（不断尝试）来获取锁，直到成功为止。

CAS

分类于 JUC

CAS（Compare-and-Swap 比较并交换） 是一种乐观锁的实现方式，是一种无锁的原子操作

一.乐观锁与悲观锁

悲观锁

悲观锁就是认为每次访问共享资源时会发生冲突，所以每次访问共享资源之前都会先加锁

乐观锁

乐观锁就是总是假设对共享资源的访问没有冲突，所以每次访问共享资源无需加锁也无需等待，当发送线程冲突的时候，乐观锁再去解决冲突比如说使用cas技术

二.什么是cas

在cas 中，有这样三个值：

  • V：要更新的变量(var)
  • E：预期值(expected)
  • N：新值(new)

比较并交换的过程如下：

判断 V 是否等于 E，如果等于，将 V 的值设置为 N；如果不等，说明已经有其它线程更新了 V，于是当前线程放弃更新，什么都不做。

这里的预期值 E 本质上指的是“旧值”。

举例说明

  1. 如果有一个多个线程共享的变量<font style="color:rgb(44, 62, 80);background-color:rgba(127, 127, 127, 0.12);">i</font>原本等于 5，我现在在线程 A 中，想把它设置为新的值 6;
  2. 我们使用 CAS 来做这个事情；
  3. 首先我们用 i 去与 5 对比，发现它等于 5，说明没有被其它线程改过，那我就把它设置为新的值 6，此次 CAS 成功，<font style="color:rgb(44, 62, 80);background-color:rgba(127, 127, 127, 0.12);">i</font>的值被设置成了 6；
  4. 如果不等于 5，说明<font style="color:rgb(44, 62, 80);background-color:rgba(127, 127, 127, 0.12);">i</font>被其它线程改过了（比如现在<font style="color:rgb(44, 62, 80);background-color:rgba(127, 127, 127, 0.12);">i</font>的值为 2），那么我就什么也不做，此次 CAS 失败，<font style="color:rgb(44, 62, 80);background-color:rgba(127, 127, 127, 0.12);">i</font>的值仍然为 2。

在这个例子中，<font style="color:rgb(44, 62, 80);background-color:rgba(127, 127, 127, 0.12);">i</font>就是 V，5 就是 E，6 就是 N。

有没有可能判断了i为5之后，正准备更新它的新值的时候，被其他线程更改了i的值？

不会，cas是一种原子操作，在cpu层面保证了它的原子性

当多个线程同时使用 CAS 操作一个变量时，只有一个会胜出，并成功更新，其余均会失败，但失败的线程并不会被挂起，仅是被告知失败，并且允许再次尝试，当然也允许失败的线程放弃操作。

三.java cas 实战

原子操作基本数据类型

基本类型的原子操作主要有这些：

  1. AtomicBoolean：以原子更新的方式更新 boolean；
  2. AtomicInteger：以原子更新的方式更新 Integer;
  3. AtomicLong：以原子更新的方式更新 Long；

这几个类的用法基本一致，这里以 AtomicInteger 为例。

  1. <font style="color:rgb(44, 62, 80);background-color:rgba(127, 127, 127, 0.12);">addAndGet(int delta)</font> ：增加给定的 delta，并获取新值。
  2. <font style="color:rgb(44, 62, 80);background-color:rgba(127, 127, 127, 0.12);">incrementAndGet()</font>：增加 1，并获取新值。
  3. <font style="color:rgb(44, 62, 80);background-color:rgba(127, 127, 127, 0.12);">getAndSet(int newValue)</font>：获取当前值，并将新值设置为 newValue。
  4. <font style="color:rgb(44, 62, 80);background-color:rgba(127, 127, 127, 0.12);">getAndIncrement()</font>：获取当前值，并增加 1。

原子操作的数组类型

如果需要原子更新数组里的某个元素，atomic 也提供了相应的类：

  1. AtomicIntegerArray：这个类提供了一些原子更新 int 整数数组的方法。
  2. AtomicLongArray：这个类提供了一些原子更新 long 型证书数组的方法。
  3. AtomicReferenceArray：这个类提供了一些原子更新引用类型数组的方法。

这几个类的用法一致，就以 AtomicIntegerArray 来总结下常用的方法：

  1. <font style="color:rgb(44, 62, 80);background-color:rgba(127, 127, 127, 0.12);">addAndGet(int i, int delta)</font>：以原子更新的方式将数组中索引为 i 的元素与输入值相加；
  2. <font style="color:rgb(44, 62, 80);background-color:rgba(127, 127, 127, 0.12);">getAndIncrement(int i)</font>：以原子更新的方式将数组中索引为 i 的元素自增加 1；
  3. <font style="color:rgb(44, 62, 80);background-color:rgba(127, 127, 127, 0.12);">compareAndSet(int i, int expect, int update)</font>：将数组中索引为 i 的位置的元素进行更新

可以看出，AtomicIntegerArray 与 AtomicInteger 的方法基本一致，只不过在 AtomicIntegerArray 的方法中会多一个数组索引 i。

原子操作的引用类型

  1. AtomicReference：原子更新引用类型；
  2. AtomicReferenceFieldUpdater：原子更新引用类型里的字段；
  3. AtomicMarkableReference：原子更新带有标记位的引用类型；

原子更新字段类型

如果需要更新对象的某个字段，atomic 同样也提供了相应的原子操作类：

  1. AtomicIntegeFieldUpdater：原子更新整型字段类；
  2. AtomicLongFieldUpdater：原子更新长整型字段类；
  3. AtomicStampedReference：原子更新引用类型，这种更新方式会带有版本号，是为了解决 CAS 的 ABA 问题，

四.CAS三大问题

尽管 CAS 提供了一种有效的同步手段，但也存在一些问题，主要有以下三个：ABA 问题、长时间自旋、多个共享变量的原子操作。

ABA问题

所谓的 ABA 问题，就是一个值原来是 A，变成了 B，又变回了 A。这个时候使用 CAS 是检查不出变化的，但实际上却被更新了两次。因为cas其实是分两步第一步是读取操作（原子），第二部才是比较和交换（原子）

ABA 问题的解决思路是在变量前面追加上版本号或者时间戳。从 JDK 1.5 开始，JDK 的 atomic 包里提供了一个类<font style="color:rgb(44, 62, 80);background-color:rgba(127, 127, 127, 0.12);">AtomicStampedReference</font>类来解决 ABA 问题。

这个类的<font style="color:rgb(44, 62, 80);background-color:rgba(127, 127, 127, 0.12);">compareAndSet</font>方法的作用是首先检查当前引用是否等于预期引用，并且检查当前标志是否等于预期标志，如果二者都相等，才使用 CAS 设置为新的值和标志。

来看参数：

  • expectedReference：预期引用，也就是你认为原本应该在那个位置的引用。
  • newReference：新引用，如果预期引用正确，将被设置到该位置的新引用。
  • expectedStamp：预期标记，这是你认为原本应该在那个位置的标记。
  • newStamp：新标记，如果预期标记正确，将被设置到该位置的新标记。

执行流程：

①、<font style="color:rgb(44, 62, 80);background-color:rgba(127, 127, 127, 0.12);">Pair<V> current = pair;</font> 这行代码获取当前的 pair 对象，其中包含了引用和标记。

②、接下来的 return 语句做了几个检查：

  • <font style="color:rgb(44, 62, 80);background-color:rgba(127, 127, 127, 0.12);">expectedReference == current.reference && expectedStamp == current.stamp</font>：首先检查当前的引用和标记是否和预期的引用和标记相同。如果二者中有任何一个不同，这个方法就会返回 false。
  • 如果上述检查通过，也就是说当前的引用和标记与预期的相同，那么接下来就会检查新的引用和标记是否也与当前的相同。如果相同，那么实际上没有必要做任何改变，这个方法就会返回 true。
  • 如果新的引用或者标记与当前的不同，那么就会调用 casPair 方法来尝试更新 pair 对象。casPair 方法会尝试用 newReference 和 newStamp 创建的新的 Pair 对象替换当前的 pair 对象。如果替换成功，casPair 方法会返回 true；如果替换失败（也就是说在尝试替换的过程中，pair 对象已经被其他线程改变了），casPair 方法会返回 false。

长时间自旋

CAS 多与自旋结合。如果自旋 CAS 长时间不成功，会占用大量的 CPU 资源。解决思路是让 JVM 支持处理器提供的pause 指令。pause 指令能让自旋失败时 cpu 睡眠一小段时间再继续自旋。

多个共享变量的原子操作

当对一个共享变量执行操作时，CAS 能够保证该变量的原子性。但是对于多个共享变量，CAS 就无法保证操作的原子性，这时通常有两种做法：

  1. 使用<font style="color:rgb(44, 62, 80);background-color:rgba(127, 127, 127, 0.12);">AtomicReference</font>类保证对象之间的原子性，把多个变量放到一个对象里面进行 CAS 操作；
  2. 使用锁。锁内的临界区代码可以保证只有当前线程能操作。

线程的创建方式有哪些

继承Thread类

这是最直接的一种方式，用户自定义类继承java.lang.Thread类，重写其run()方法，run()方法中定义了线程执行的具体任务。创建该类的实例后，通过调用start()方法启动线程。

实现Runnable 接口

如果一个类已经继承了其他类，就不能再继承Thread类，此时可以实现java.lang.Runnable接口。实现Runnable接口需要重写run()方法，然后将此Runnable对象作为参数传递给Thread类的构造器，创建Thread对象后调用其start()方法启动线程。

实现Callable接口与FutureTask

Callable 与 Runnable 类似，但 call() 方法可以返回结果，并且允许抛出受检异常（因为签名是 V call() throws Exception）。

Runnable.run() 和 Thread.run() 只能抛运行时异常，且没有返回值（签名为 void run()），因此无法处理受检异常，也无法产生结果。

如果要执行 Callable，必须将其包装为 FutureTask，再交给线程或线程池执行。

工厂模式分类

模式类型 核心思想 结构特点 优点 缺点 适用场景
简单工厂（Simple Factory） 由一个工厂类集中创建所有产品 一个工厂类 + 多个产品类 简单易用，统一创建入口 新增产品需修改工厂代码，不符合开闭原则 产品种类少、变化不频繁

抽象工厂（Abstract Factory） 生产一组相关或依赖的产品族 抽象工厂接口 + 多个具体工厂 + 多个产品族 可一次创建多个配套产品，保持一致性 难以扩展新产品族（需改接口） 需要创建多个“配套产品”的系统（如主题样式、品牌设备）

工厂方法是“一个工厂生产一种产品”；抽象工厂是“一个工厂生产一整套相关产品”。

分类于 操作系统

什么是DMA

DMA（Direct Memory Access）允许 I/O 设备在 CPU 不参与数据搬运的情况下，直接访问内存，实现“设备 ↔ 内存”的高速传输。
CPU 只负责发起和结束，不负责搬运。

什么是零拷贝

零拷贝技术让数据在内核态内部直接流动，由 DMA 完成搬运，避免用户态与内核态之间的数据复制，从而减少 CPU 开销并提升 I/O 性能。

比如再进行网络io的时候，先来看没有零拷贝时的io方式

传统 I/O 中既有 4 次用户态与内核态的切换，也有 4 次数据拷贝；如果只是单纯搬运数据，这些操作会造成巨大开销，因此提升性能的关键就是减少上下文切换次数和内存拷贝次数。

虚拟内存

分类于 操作系统

什么是虚拟内存

虚拟内存 = 让每个进程看到一片连续的大地址空间，并由操作系统自动把它映射到真实、碎片化的物理内存中，从而解决内存不够用、内存碎片和进程隔离的问题。

虚拟内存主要有两种形式一种是内存分段（早期提出，现在很少使用），一种是内存分页（目前普遍流行的）

内存分段

虚拟内存 = 让每个进程看到一片连续的大地址空间，并由系统自动把它映射到真实、碎片化的物理内存中，从而解决内存不够用、内存碎片和进程隔离的问题。

分段划分，本质上就是通过一个段表进行虚拟地址->真实地址的映射

分段划分会有两个问题

  • 内存碎片
  • 内存交换的效率低

内存碎片

我们来看看这样一个例子。假设有 1G 的物理内存，用户执行了多个程序，其中：

  • 游戏占用了 512MB 内存
  • 浏览器占用了 128MB 内存
  • 音乐占用了 256 MB 内存。

这个时候，如果我们关闭了浏览器，则空闲内存还有 1024 - 512 - 256 = 256MB。

如果这个 256MB 不是连续的，被分成了两段 128 MB 内存，这就会导致没有空间再打开一个 200MB 的程序。

分段会产生“外部碎片”，不会产生“内部碎片”。

  • 不会产生内部碎片：
    分段按“真实需求大小”整体分配，只有找到能容纳整个段的连续空间才会分配，因此不会出现“分配多余空间被浪费”的情况。
  • 会产生外部碎片：
    由于段大小不固定、必须连续，内存中会形成许多小的、彼此不连续的空洞，可能导致后续大的段无法放入，进而产生外部碎片。

对于内存分段的外部内存碎片问题，可以使用内存交换来解决

可以把音乐程序占用的那 256MB 内存写到硬盘上，然后再从硬盘上读回来到内存里。不过再读回的时候，我们不能装载回原来的位置，而是紧紧跟着那已经被占用了的 512MB 内存后面。这样就能空缺出连续的 256MB 空间，于是新的 200MB 程序就可以装载进来。

内存交换效率低

由于内存分段很容易产生外部内存碎片，所以我们不得不进行大量的内存交换，而内存交换涉及到磁盘io效率很低

内存分页

分页是把整个虚拟和物理内存空间切成一段段固定尺寸的大小。这样一个连续并且尺寸固定的内存空间，我们叫页（Page）。在 Linux 下，每一页的大小为 <font style="color:rgb(71, 101, 130);background-color:rgba(27, 31, 35, 0.05);">4KB</font>。虚拟地址与物理地址之间通过页表来映射，如下图：

页表是存储在内存里的，内存管理单元 （MMU（计算机硬件））就做将虚拟内存地址转换成物理地址的工作。

分页是怎么解决分段的「外部内存碎片和内存交换效率低」的问题？

1. 分页不会有外部碎片
内存被预先划分成大小固定的页（如 4KB），页与页紧密排列，因此不会出现“不连续的小空洞”。

2. 分页会有内部碎片
内存分配的最小单位是页，即使程序只用到几百字节，也必须分配整页，所以会产生内部碎片。

3. 分页的换入换出效率更高
当内存不足时，操作系统只需将“近期未使用的少量页面”换出到磁盘，再按需将这些页面换入，而不是移动整个大段，因此交换开销更小。

4. 分页支持按需加载
程序启动时无需一次性加载所有内容，只需建立虚拟页与物理页的映射，程序运行到需要某个虚拟页时，再触发缺页中断并把对应页加载到内存即可。

分页机制下，虚拟地址和物理地址是如何映射的？

在分页机制下，虚拟地址分为两部分，页号和页内偏移。页号作为页表的索引，页表包含物理页每页所在物理内存的基地址，这个基地址与页内偏移的组合就形成了物理内存地址，见下图。

![](https://cdn.nlark.com/yuque/0/2025/png/50543647/1763612066346-bc916948-fa43-48be-8fe2-b1a0ff5cb937.png)

总结一下，对于一个内存地址转换，其实就是这样三个步骤：

  • 把虚拟内存地址，切分成页号和偏移量；
  • 根据页号，从页表里面，查询对应的物理页号；
  • 直接拿物理页号，加上前面的偏移量，就得到了物理内存地址。

多级页表

对于只有一个页表的简单分页

  • 每个进程都有独立的虚拟地址空间，因此每个进程都需要自己的页表。
  • 以 32 位系统为例：虚拟空间 4GB、页大小 4KB → 需要约 100 万个页。
  • 每个页表项 4 字节 → 一个完整页表大小约 4MB。
  • 一个进程 4MB 看似不大，但 100 个进程就要 400MB 内存仅用于存页表。
  • 在 64 位环境下虚拟地址空间更大，页表会更巨大。

为了解决空间问题，就引入了多级页表

对于32位操作系统我们把这个 100 多万个「页表项」的单级页表再分页，将页表（一级页表）分为 <font style="color:rgb(71, 101, 130);background-color:rgba(27, 31, 35, 0.05);">1024</font> 个页表（二级页表），每个表（二级页表）中包含 <font style="color:rgb(71, 101, 130);background-color:rgba(27, 31, 35, 0.05);">1024</font> 个「页表项」，形成二级分页。如下图所示：

分了二级表，映射 4GB 地址空间就需要 4KB（一级页表）+ 4MB（二级页表）的内存，这样占用空间不是更大了吗？

  • 虽然虚拟地址空间是 4GB，但进程实际只使用其中很小的一部分，所以绝大多数页表项始终是空的。
  • 如果只使用一级页表，就必须一次性构建覆盖整个 4GB 空间的页表（约 4MB），而且每个进程都要有一份，这会造成巨大浪费。
  • 二级页表的做法是：一级页表仍覆盖整个虚拟空间，但只有当某个一级页表项被访问时，才按需创建对应的二级页表。
  • 一级页表始终存在（大小约 4KB）
  • 二级页表“用到哪一块虚拟空间就创建哪一块”，未访问的区域不占用页表内存

对于64位操作系统我们一般会使用四级页表

为什么不分级的简单页表不能按需创建？

因为简单页表承担的是“直接把虚拟页号映射到页表项”的工作，它必须完整存在，否则 MMU 无法完成地址转换。 所以页表一定要覆盖全部虚拟地址空间

TLB

根据局部性原理，程序在短时间内会频繁访问相同的地址区域。所以在cpu 芯片中加入了一个专门存放程序最常访问的页表项的 Cache，这个 Cache 就是 TLB（Translation Lookaside Buffer） ， 通常称为页表缓存、转址旁路缓存、快表等。

CPU 内部集成了内存管理单元（MMU），负责地址转换，并与 TLB 协同工作。

有了 TLB 后，CPU 查地址时会先查 TLB；如果命中，则无需访问页表；如果未命中，再去查正常的页表。

由于程序具有局部性，常访问的页通常只有少数几个，所以 TLB 的命中率往往非常高。

一致性hash

分类于 分布式

什么是一致性hash

假设有一个KV（key-value）缓存系统，某个key应该到哪个或者哪些节点上获得，应该是确定的。

因此我们可以使用hash算法，如果客户端要获取指定 key 的数据，通过下面的公式可以定位节点：

hash(key) % 3


但是有一个很致命的问题，如果节点数量发生了变化，也就是在对系统做扩容或者缩容时，必须迁移改变了映射关系的数据，否则会出现查询不到数据的问题。

一致性哈希算法就很好地解决了分布式系统在扩容或者缩容时，发生过多的数据迁移的问题。

一致哈希算法也用了取模运算，但与哈希算法不同的是，哈希算法是对节点的数量进行取模运算，而一致哈希算法是对 2^32（不一定，环的大小可以任意） 进行取模运算，是一个固定的值。

一致性哈希要进行两步哈希：

  • 第一步：对存储节点进行哈希计算，也就是对存储节点做哈希映射，比如根据节点的 IP 地址进行哈希；
  • 第二步：当对数据进行存储或访问时，对数据进行哈希映射；

接着，对要查询的 key-01 进行哈希计算，确定此 key-01 映射在哈希环的位置，然后从这个位置往顺时针的方向找到第一个节点，就是存储该 key-01 数据的节点。

假设节点数量从3增加到4，新的节点D经过hash计算后映射到了下图中的位置

你可以看到，key-01、key-03 都不受影响，只有 key-02 需要被迁移节点 D。

假设节点数量从 3 减少到了 2，比如将节点 A 移除：

你可以看到，key-02 和 key-03 不会受到影响，只有 key-01 需要被迁移节点 B。

因此，在一致哈希算法中，如果增加或者移除一个节点，仅影响该节点在哈希环上顺时针相邻的后继节点，其它数据也不会受到影响。

一致性hash 有什么问题

存在节点分布不均匀的问题，如下图

如何解决

一致性 Hash 引入 虚拟节点，为每个真实节点创建多个映射点，使节点在 Hash 环上的分布变得更均匀，从而解决真实节点少时分布不均的问题，并提高系统在节点增删时的稳定性。

为什么需要虚拟节点？

真实节点太少时，映射到哈希环上的位置不均匀，导致：

  • 请求落点集中在少数节点 → 负载不平衡
  • 节点上下线时，影响区间太大 → 系统不稳定

真实节点少时的“一次性分布不均”问题。
