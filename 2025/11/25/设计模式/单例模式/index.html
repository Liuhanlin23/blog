单例模式

分类于 设计模式

单例模式的核心概念就是一个类在整个程序运行过程中只能有一个实例，并且提供一个全局访问点

适用场景

当系统中 某个类只需要一个实例，并且该实例需要被全局共享或频繁访问时，就应该使用单例模式。

比如：

场景 说明 示例
配置管理类 系统配置信息只需加载一次，全局共享，避免重复读取和资源浪费。 ConfigManager
、Spring Environment
日志管理类 统一日志输出入口，防止多个实例竞争文件，保证日志写入顺序一致性。 Logger
（如 Log4j
、SLF4J
）
框架核心组件 框架的控制中心、调度器等组件需要全局唯一实例，以便统一管理与访问。 Spring ApplicationContext
、ServletContext

为什么要加volatile

INSTANCE = new DclSingleton(); 实际上分为三步指令：

  • 分配内存
  • 调用构造函数初始化对象
  • 将引用赋值给变量 INSTANCE

如果 发生指令重排（比如执行顺序 1 → 3 → 2），
其他线程可能读取到一个还没初始化完成的对象（即“半初始化对象”），导致 发生错误。

volatile 关键字可以：

  • 禁止指令重排
  • 保证多线程间的可见性
  • 确保对象创建的安全性与完整性。

双重检查和直接加锁相对于直接加锁有什么优势呢？

直接加锁代码实现

注意看直接加锁的代码不论是否是第一次创建都会上锁，所以会造成并发能力的下降，而双重检查是只有当第一次创建的时候才会上锁，创建完成之后访问的话不需要加锁，并发能力大大增强

synchronized (DclSingleton.class) vs synchronized (this) 的根本区别

对比项 synchronized (DclSingleton.class) synchronized (this)
锁对象 类对象（Class 对象） 当前实例对象（this）
锁的作用范围 整个类层面（所有实例共享同一把锁） 仅当前实例有效（不同实例不同锁）
是否可用于静态方法 ✅ 是（静态方法中没有 this） ❌ 否（静态方法中没有 this）
线程安全级别 保证类级别线程安全（全局唯一锁） 仅保证单个对象的线程安全
使用场景 单例模式、静态方法、全局资源同步 对象方法、实例级别同步

单例模式中的getInstance（）是一个静态方法，属于类方法，因此方法中没有this，所以不能用this加锁

也就是说：

在单例模式中，用 synchronized (DclSingleton.class) 才能真正保证全局唯一实例的线程安全。
如果用 synchronized(this)，此时连实例都还没创建，根本拿不到 this，也无法锁。
