什么时候需要 不需要创建索引

分类于 MySQL ， 索引

一.什么时候需要创建索引

  • 唯一性字段：如商品编码、身份证号等。保证唯一、方便定位。
  • 常用于 WHERE 的字段：提高查询过滤效率；若有多个条件，可建联合索引。
  • 常用于 GROUP BY / ORDER BY 的字段：索引中的数据有序，可避免额外排序，提升分组与排序性能。

理解为什么 GROUP BY / ORDER BY 字段使用索引可以避免额外排序

MySQL 的 B+Tree 索引 是有序结构，索引中的键值天然按升序（或定义顺序）存储。因此：

  • 如果查询排序字段或分组字段使用了索引，MySQL 可直接按照索引顺序扫描数据。
  • 无需再在内存中执行排序操作，性能更高。

三种典型情况

情况一：单列索引可直接排序

若 age 有索引，索引中的记录已按 age 升序存放。
MySQL 直接顺序扫描即可，无需再次排序。

✅ 无需额外排序。

情况二：WHERE 和 ORDER BY 使用不同字段

WHERE 使用 age 索引过滤，但 name 并非同一索引顺序，
筛选结果仍需在内存中重新排序。

⚠️ 需要额外排序（filesort）。

情况三：联合索引同时覆盖过滤与排序

联合索引 (age, name) 既可加速 WHERE，又保持 ORDER BY 顺序一致。

✅ 完全利用索引，一次扫描即可输出有序结果。

总结

  • 索引的有序性使得 GROUP BY / ORDER BY 能直接使用索引顺序。
  • WHERE + ORDER BY 若在同一索引上，可实现“边查边有序”。
  • 不同字段或索引不匹配时，MySQL 必须额外排序，性能下降。

二. 什么时候不需要创建索引

1. 不参与查询条件的字段

  • 在 WHERE、GROUP BY、ORDER BY 中从未出现的字段没必要建索引；
  • 索引只对“查找、排序、分组”起作用，其他字段建了也白搭。

2. 重复值太多的字段

  • 如性别（男女）或状态（0/1），区分度太低；
  • MySQL 查询优化器会自动判断这些情况并放弃索引，直接全表扫描。

3. 表数据太少

  • 数据量几百上千行以内，索引带来的维护开销 > 性能收益。

4. 高频更新的字段

  • 比如用户余额、库存数量等频繁变化字段；
  • 每次更新都要维护 B+Tree，有序性调整会拖慢性能。

一句话总结

不查、不分、不排、重复多、数据少、改太勤 → 都别建索引。
