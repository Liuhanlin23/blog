分类于 网络

一.HTTP与HTTPS的区别

对比项 HTTP HTTPS
全称 HyperText Transfer Protocol HyperText Transfer Protocol Secure
端口号 80 443
是否加密 ❌ 明文传输，容易被窃听或篡改 ✅ 加密传输（TLS/SSL）保证机密性和完整性
安全机制 无 使用 SSL/TLS 协议
证书 不需要 需要 数字证书（CA颁发） 验证服务器身份
性能 快一些（无加解密） 稍慢一点（有加解密开销）但可用硬件/协议优化（如 HTTP/2）
数据完整性 无校验机制 有完整性校验（防止被篡改）
典型用途 一般网站、内部服务 登录、支付、银行、电商、隐私相关业务

二.HTTPS的加密过程

https只是在http协议和tcp层之间加了一个TLS(原SSL)的安全传输协议

TLS工作原理

TLS 的核心思想：结合对称加密与非对称加密，兼顾安全与效率。

  • 对称加密：同一密钥加解密，效率高但密钥传输不安全。
  • 非对称加密：公钥加密、私钥解密，安全但效率低。
    → TLS 用“非对称加密”安全地传输“对称密钥”，再用“对称加密”进行高速通信。

基本流程

  1. 服务器 生成一对非对称密钥（公钥 + 私钥）。
  2. 公钥 通过数字证书下发给客户端。
  3. 客户端 生成随机的对称密钥，用服务器公钥加密后发送给服务器。
  4. 服务器 用私钥解密，得到相同的对称密钥。
  5. 之后双方使用该对称密钥进行加密通信。

存在的问题

若公匙直接明文传输公钥，可能被中间人伪造。
因此 TLS 通过 数字证书（CA 签名） 验证公钥的真实性，防止中间人攻击。

为什么明文传输公匙会有中间人攻击

假设攻击者在中间拦截了你和服务器之间的通信：

你 ←→ 攻击者 ←→ 服务器

他可以：

  1. 拦截服务器发来的公钥；
  2. 换成自己生成的一对公钥/私钥，发给你；
  3. 你以为是服务器的公钥，实际是攻击者的；
  4. 之后你发的加密数据，攻击者能用自己的私钥解密；
  5. 他再用服务器公钥重新加密转发——你和服务器都没发现被劫持。

这就是典型的中间人攻击

CA证书

CA 证书是什么

CA (Certificate Authority) 是数字证书，用来证明网站的身份真实性，防止公钥在传输中被篡改或伪造。

工作原理

证书签发过程

  1. 网站生成一对密钥（公钥 + 私钥）。
  2. 将公钥 + 域名等信息打包成 CSR 申请文件 ，提交给 CA。
  3. CA 验证网站身份（确认确实是 bank.com）。
  4. CA 使用 CA 私钥 对证书内容的 hash 摘要 进行加密，生成 数字签名。
    ✅ 生成的证书包括：网站公钥、域名信息、CA 签名。

注意： 签名只是对 hash 摘要加密，而不是加密整个证书。

浏览器的验证过程

  1. 服务器把证书发给浏览器。
  2. 浏览器提取证书中的：
    • 公钥信息
    • CA 的数字签名
  3. 浏览器使用 CA 公钥 (系统内置) 解密签名，得到摘要 A。
  4. 再对证书明文部分重新做 hash，得到摘要 B。
  5. 比较 A 和 B：
    • 相同→ 证书未被修改，是合法的
    • 不同→ 证书被篡改，拒绝连接。
